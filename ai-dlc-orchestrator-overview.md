# ai-dlc-orchestrator 소개 — 목적·구성·사용법

> **한 줄 정의.** ai-dlc-orchestrator는 *여러 레포에 걸친 AI 주도 개발*을, 자연어 지시 하나로 조율하는 **사내 공유 깃 룰북**이다.

---

## 1. 왜 만드나 (목적)

**문제.** AI 코딩 도구는 대개 *레포 하나* 안에서만 잘 동작한다. 그런데 실제 제품은 프론트·백엔드·인프라 등 *여러 레포*에 걸쳐 있고, 그 사이의 인터페이스·의존 관계·통합·일관성은 사람이 직접 챙겨야 한다. AI에게 "이 기능 만들어줘"라고 하면 한 레포는 잘 만들어도, 레포 경계를 넘는 조율은 빠진다.

**해법.** ai-dlc-orchestrator는 그 *시스템 레벨 조율*을 담당한다. 사용자는 자연어로 한 번 지시하고, 시스템이:
1. 어느 레포들이 연관되는지 **분해**하고,
2. 레포 사이의 **계약(인터페이스)을 설계**하고,
3. 각 레포의 실제 구현은 검증된 오픈소스에 **위탁**하고,
4. 결과를 **통합·테스트·커밋**하고, 나아가 **CI/CD 부착 → 릴리스 → 환경 배포(검증·롤백)** 까지 한 사이클로 끌고 간다 — 설계부터 배포 착지까지 end-to-end.

**핵심 이점.**
- 멀티 레포 조율 자동화 — 사람은 핵심 결정만 승인.
- 레포 내부 구현은 오픈소스(AWS AI-DLC)에 위임 → **AWS 계정 불필요, 비용 0** (MIT-0 라이선스).
- 모든 룰이 깃 리포 안의 마크다운 → 버전 관리·PR로 투명하게 진화. 블랙박스가 아니다.

## 2. 핵심 아이디어 (구조)

### 2-레벨 구조

```
        사용자 (고객)
           │  자연어 지시 한 번
           ▼
   오케스트레이터 (팀장)          ← ai-dlc-orchestrator · 시스템 레벨 · STEP 0~10
           │  분해 · 설계 후 레포별 위탁
   ┌───────┼───────┐
   ▼       ▼       ▼
 레포 A   레포 B   레포 C         ← 각 레포 안의 구현은
 [AWS]    [AWS]    [AWS]            AWS AI-DLC (팀원)가 수행
```

- **시스템 레벨 = ai-dlc-orchestrator (우리).** 멀티 레포 분해·cross-repo 설계·통합·테스트·커밋 조율.
- **레포 레벨 = AWS `awslabs/aidlc-workflows` (오픈소스).** 단일 레포 안에서 요구사항 → 설계 → 코드 → 테스트 → 빌드.

즉 ai-dlc-orchestrator는 *직접 코딩하는 도구*가 아니라, 레포별 AI 워크플로우를 **지휘하는 상위 레이어**다.

### 멘탈 모델 — 팀 비유

| 역할 | 대응 |
|---|---|
| 사용자 | 고객 |
| 오케스트레이터 | 팀장 — 사용자는 팀장하고만 대화 |
| 서브에이전트 | 팀원 — 팀장을 통해서만 움직임 |

사용자는 창구 하나(팀장)만 상대하면 된다. 나머지 협업은 내부에서 처리된다.

## 3. 무엇을 만들고 있나 (구성물)

ai-dlc-orchestrator 깃 리포 하나. 안의 모든 것이 마크다운 룰 파일이다.

- **`agents/` — 행위 주체(에이전트) 룰북**
  - `orchestrator/` — 팀장. 인텐트 분류·라우팅·의사결정·예외·종료 (코어 + 디테일 4개)
  - `SETTER` — 시스템 최초 부트스트랩
  - `REPO-SETTER` — 각 레포에 AWS 룰셋 + 필요한 Extension 설치
  - `REPO-CREATOR` — 새 레포 생성
  - `CICD-SETTER` — 레포별 CI/CD 부착·통합 (기존 탐지 / 신규 생성, 조건부)
  - `DEPLOY-AGENT` — 적응형 환경 배포 실행 (non-prod 자율 / prod 사람 게이트, 조건부)
  - `CYCLE-CLOSER` — 사이클 종료 후처리 (기록 최종화·보고)
  - `HANDOFF-WRITER` — 세션 이관 문서 자동 작성
- **`specs/` — 형식·룰 명세 (에이전트 아님)**
  - `SYSTEM-WORKFLOW` — STEP 0~10 시스템 워크플로우 본문 + §3 공통 규칙(fetch 선행·공통 정책 POLICY-*)
  - `AWS-ADAPTER` — AWS AI-DLC 연동 인터페이스 (STEP↔AWS 매핑·버전 핀·산출물 흡수)
  - `CYCLE-LOG` — 작업 기록(audit) 표준
  - `EVOLUTION` — DP-9 진화 휴리스틱 본문 (사용 패턴 감지 → 룰북 갱신 능동 제안)
  - `VERIFICATION` — 검증·지상검증 규율(POLICY-VERIFY) — "성공/완료" 클레임을 지상 실제로 확인하는 횡단 규율
  - `CICD-RELEASE-ADAPTER` — CI/CD·릴리스 머신 통합(POLICY-RELEASE) — 게이트 인식·사람 승인 게이트·릴리스 플로우 (레포가 보유 시에만 발동하는 조건부 명세)
  - `DEPLOY-ADAPTER` — 환경 배포 룰(POLICY-DEPLOY) — 환경 모델·전략·prod 사람 게이트·health 지상검증·롤백 (실제 배포 시에만 발동하는 조건부 명세)
- **`templates/` — 채워서 인스턴스를 찍어내는 원본**
  - 시스템·핸드오프 템플릿: `ORCHESTRATOR` / `REPO-MAP` / `HANDOFF`
  - Stage-1 레포 템플릿 7종: `CLAUDE` / `STACK` / `WORKFLOW` / `DESIGN` / `CHECKLIST` / `CODING` / `FRAMEWORK`
  - `extensions/` — 레포에 선택 적용하는 룰: 품질 체크리스트(quality), 코딩 규칙(coding), 프레임워크 패턴(framework)

### 진행 단계

| 단계 | 내용 | 상태 |
|---|---|---|
| Stage 1 | 단일 레포용 템플릿 7종 | ✓ 완료 |
| Stage 2 | 시스템 부트스트랩 + 오케스트레이터 + AWS 통합 | ✓ 완료 |
| Stage 3 | EVOLUTION — 사용 패턴을 학습해 룰북 진화 능동 제안 | ✓ 본문 완료 (`specs/EVOLUTION.md` · 운영 중 패턴 누적은 지속) |
| Stage 4 | 협업 거버넌스 — 공유 `dlc-meta` 원격 · 자격증명 경계 · 공유 산출물 추적 | ✓ 완료 (POLICY-TRACKING · 머신독립 식별자) |

> **현재 단계:** 외부 공유 퍼블리시를 위한 **범용화·하드닝** — 어떤 프로젝트(OS·스택 무관)에서도 부트스트랩이 정상 동작하도록 공통 정책(POLICY-ENCODING/TEMPLATE-ADHERENCE/TRACKING) 도입, 검증 프로토콜·CI/릴리스·디자인 어댑터 보강. 이제 **CI/CD 부착 → 릴리스 → 환경 배포(검증·롤백)** 까지 더해, *설계→배포* 풀 라이프사이클을 end-to-end로 커버한다.

## 4. 어떻게 쓰나 (사용 흐름)

### 준비 (1회)

1. `ai-dlc-orchestrator` 리포 클론
2. `SETTER`가 시스템 부트스트랩 (오케스트레이터·레포 맵 초기화)
3. `REPO-SETTER`가 대상 레포마다 AWS 룰셋 + 필요한 Extension 설치

### 일상 사용

사용자가 오케스트레이터에게 **자연어로 지시** → 인텐트 유형 자동 분류 → 그에 맞는 STEP 경로 실행.

인텐트 유형 예: *새 기능 개발*(STEP 0~10 풀 사이클), *변경·수정*(STEP 1~9), *분석·조사*(STEP 1~2, 코드 변경 없이 보고), 그 외 *추가 레포 필요 / 사이클 종료 / 시스템 부트스트랩 / 진화·메타*.

### 한 사이클 (STEP 0~10) 요약

| STEP | 하는 일 |
|---|---|
| 0 | 인텐트 수신·분류 |
| 1~3 | 멀티 레포 분해 → cross-repo 설계 → 사용자 피드백 |
| **4** | ★ 각 레포 구현을 **AWS AI-DLC에 위탁** |
| 5~7 | 통합 → 통합 테스트 → 시스템 빌드 |
| 8~9 | (필요 시) 보조 문서 → 다중 레포 커밋·릴리스 |
| **9.5** | *(조건부)* 환경 배포 착지 — `DEPLOY-AGENT` (non-prod 자율 / prod 사람 게이트) · 배포 후 검증·롤백. STEP 재번호 아님, 9와 10 사이 조건부 활동 |
| 10 | 사이클 로그 → 진화 입력 |

**사람의 개입.** 모든 단계 승인이 아니라, *위험·큰 영향·비가역* 지점만 골라서 확인하는 **선택적 게이트**. 나머지는 AI가 자율 진행한다.

### 예시

> 사용자: "결제 모듈에 쿠폰 기능 추가해줘"

→ 오케스트레이터가 *결제·주문·프론트* 레포가 영향받는다고 분해
→ 레포 간 API 계약 설계 → 사용자 한 번 확인
→ 각 레포 구현을 AWS에 위탁 → 결과 통합·테스트 → 커밋.

사용자는 **설계를 한 번 확인하고 결과를 받는다.**

---

## 5. 한 줄 요약

멀티 레포 AI 개발의 **"팀장" 역할**을 하는, 깃으로 관리되는 **오픈소스 기반 조율 룰북.** 사용자는 자연어로 지시하고 핵심 결정만 승인하면, 시스템이 레포 분해부터 통합·커밋까지 한 사이클로 끌고 간다.
