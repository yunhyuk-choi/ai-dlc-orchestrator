# REPO-CREATOR.md — 추가 레포 생성 서브 에이전트 룰북

> 시스템 운영 중 DP-7(추가 레포 필요 판단) 시 오케스트레이터가 호출하는 서브 에이전트.
> 핵심: 신규 레포 디렉토리·git 생성 + REPO-MAP 갱신. REPO-SETTER 호출은 *오케스트레이터 책임*.
> 위치: `ai-dlc-orchestrator/agents/REPO-CREATOR.md`

---

## 0. 컨텍스트

본 에이전트는 시스템 운영 중 *신규 레포 필요성 발견* 시 호출:

```
[운영 중 — DP-7 추가 레포 필요 판단]
    ↓
[사용자 컨펌 — 항상 (DECISIONS.md DP-7 컨펌 정책)]
    ↓
[REPO-CREATOR 호출]
    ├─ 신규 레포 메타 인터뷰 (이름·경로·도메인·의존)
    ├─ 디렉토리 생성 + git init
    ├─ REPO-MAP.md 신규 행 append (원칙 6 단일 원천 갱신)
    └─ 오케스트레이터에 보고 (REPO-SETTER 호출 권고)
    ↓
[오케스트레이터에 4-튜플 보고]
    ↓
[오케스트레이터가 REPO-SETTER 호출 — Layer 2 셋업]
```

`{공유리포}` = 클론된 `ai-dlc-orchestrator/` 디렉토리 절대 경로.
`{메타 레포}` = `dlc-meta/` (SETTER가 부트스트랩한 위치).

## 3. 핵심 원칙

- **원칙 1**: 호출 시점·책임 분리 — 추가 레포 생성은 *드물고 특정 시점*. 부트스트랩(SETTER)·셋업(REPO-SETTER)과 분리.
- **원칙 4**: AI 자율 실행, 사용자는 의미 결정만 — 인터뷰로 이름·도메인 결정
- **원칙 6**: 단일 원천 — *REPO-MAP은 도메인 정보의 단일 원천*. REPO-CREATOR가 갱신 책임 보유. 다른 곳에 도메인 정보 중복 X.
- **원칙 7**: 오케스트레이터 호출 책임 — REPO-SETTER 호출은 *오케스트레이터가 직접* (REPO-CREATOR가 호출 X)
- **원칙 8**: 공유 룰북·템플릿은 깃 단일 원천 — `REPO-MAP.template.md` 변경은 PR/머지

| 사용자 인터뷰 응답 | RC1.x ~ RC4.x (§5.2 인터뷰 결과) |

### 4.2 출력 (오케스트레이터에 보고 — 4-튜플 표준)


### RC0 — 사전 조건 체크 (자동, 인터뷰 없음)

- `{공유리포}/templates/REPO-MAP.template.md` 존재 확인
- `{메타 레포}/REPO-MAP.md` 존재 + 쓰기 가능 확인
- 디렉토리 생성 권한 확인
- 실패 시 오케스트레이터에 보고 후 종료

### RC1 — 신규 레포 위치·이름 인터뷰

| ID | 질문 | 기본/추천 |
|---|---|---|
| RC1.1 | 신규 레포 슬러그 (URL-safe 식별자)? | DP-7 결정 시 사용자 제안 또는 도메인 기반 추천 |
| RC1.2 | 신규 레포 로컬 절대 경로? | 기존 레포들의 *상위 폴더* 기본 (`.env` `{WORKSPACE_ROOT}`) |
| RC1.3 | `git init` 진행할까? | yes |
| RC1.4 | **원격 생성 위치/네임스페이스?** (GitLab 그룹 등) | 기존 레포들과 동일 네임스페이스 |

### RC2 — 역할·도메인 인터뷰

| ID | 질문 | 자동 추천 |
|---|---|---|
| RC2.1 | 레포 역할 한 줄 요약? | DP-7 트리거 이유에서 자동 추출 |
| RC2.2 | 담당 도메인 (멀티, 콤마 구분)? | DP-7 결정 시 식별된 신규 도메인 |

### RC3 — 의존 관계 인터뷰

| ID | 질문 | 자동 추천 |
|---|---|---|
| RC3.1 | 의존하는 다른 레포 슬러그 (멀티)? | DP-7 영향 분석 결과에서 추출 (있을 시) |
| RC3.2 | 다른 레포 *역의존* 갱신 필요? | 의존 그래프 분석 결과 (있을 시) — REPO-MAP의 기존 행 갱신 필요성 |

> 역의존 갱신 시 — *기존 행 수정*은 본 에이전트 책임. RC6에서 처리.

### RC4 — 추가 메타 (선택)

| ID | 질문 | 디폴트 |
|---|---|---|
| RC4.1 | 본 사이클 안에서 즉시 REPO-SETTER 진행? | yes — 권고 |

> *no* 시 — 신규 레포 디렉토리·REPO-MAP만 생성. REPO-SETTER는 사용자 명시 시점에 진행. *yes* 시 — 권고 다음 단계에 명시.

### RC5 — 디렉토리 + git 생성 + 원격 생성·push

> **전제**: DP-7 사용자 컨펌 *이미 완료* (항상 — DECISIONS DP-7). 원격 생성도 이 컨펌 범위. 자격증명은 `.env`에서 읽음 (발급·접근제어 변경은 사람 — D-Stage4-2).

```bash
# 1. 로컬 디렉토리 + git init
mkdir -p {RC1.2}/{RC1.1}
cd {RC1.2}/{RC1.1}
[ RC1.3 == yes ] && git init

# 2. 원격 생성 (네임스페이스 {RC1.4}) — .env GITLAB_TOKEN 사용. 빈 원격이면 연결만.
#    (사용자 선결로 빈 원격 만들어둔 경우 생성 스킵하고 URL만 연결)
git remote add origin {RC1.4}/{RC1.1}.git

# 3. 초기 push (.env는 대상 레포에 없음 — dlc-meta 전용)
git add -A && git commit -m "init {RC1.1}" && git push -u origin HEAD
```

> 원격 URL(`{RC1.4}/{RC1.1}.git`)은 RC6에서 REPO-MAP `원격` 컬럼에 기록 (머신 독립 식별자).

### RC6 — REPO-MAP.md 갱신

**원칙 6 — REPO-MAP이 도메인 정보 단일 원천이므로 *반드시 갱신*.**

- 원본 형식: `{공유리포}/templates/REPO-MAP.template.md` (`{REPO_TABLE}` 변수의 행 형식 참조)
- 갱신 대상: `{메타 레포}/REPO-MAP.md`

**신규 행 append** (REPO_TABLE 형식):

| 슬러그 | 원격 | 역할 | 도메인 | 의존 |
|---|---|---|---|---|
| RC1.1 | {RC1.4}/{RC1.1}.git | RC2.1 | RC2.2 (콤마) | RC3.1 (콤마) |

**역의존 갱신** (RC3.2 yes 시):
- 기존 행의 *의존* 컬럼에 신규 슬러그 추가
- 기존 행 본문은 그 외 *수정 X* — 슬러그·경로·역할·도메인 변경 없음

### RC7 — 정합성 자체 체크 + 오케스트레이터 보고

**자체 체크 항목**:
- 신규 레포 디렉토리 실제 존재
- (RC1.3 yes 시) `.git/` 존재
- REPO-MAP.md 신규 행 추가됨 — 슬러그·원격·역할·도메인·의존 모두 채워짐
- REPO-MAP.md 슬러그 중복 없음 (신규 슬러그가 기존과 충돌하지 않음)
- 의존 컬럼이 *존재하는 슬러그*만 가리킴
- 역의존 갱신 (해당 시) 정확

**오케스트레이터 보고 (4-튜플)**:
1. 생성·수정 경로 목록
2. 정합성 체크 결과
3. 인터뷰 응답 원본 (RC1.x ~ RC4.x)
4. **권고 다음 단계** — *REPO-SETTER 호출 권고* (구체 명시):
   ```
   다음 단계: REPO-SETTER 호출
   - 대상 레포 경로: {RC1.2}/{RC1.1}
   - 슬러그: {RC1.1}
   - 도메인: {RC2.2}
   - 의존: {RC3.1}
   - RC4.1 응답: {yes / no} — 즉시 진행 여부
   ```

### RC8 — 종료

- 컨텍스트 해제
- 오케스트레이터가 RC4.1 응답에 따라 REPO-SETTER 호출 분기 결정
