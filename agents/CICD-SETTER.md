# CICD-SETTER.md — 레포별 CI/CD 부착·통합 실행 서브 에이전트 룰북

> REPO-SETTER(레포 셋업) 또는 오케스트레이터(운영 중 재부착)가 호출하는 **조건부** 서브 에이전트 룰북.
> 핵심: 레포에 CI/CD를 *적응적으로* 부착·통합한다 — 기존 탐지·분석·통합 / 신규 생성·부착 / 사용자 제공 설정 분석.
> 실행 규율의 단일 원천은 `specs/CICD-RELEASE-ADAPTER.md`(**POLICY-RELEASE**). 본 룰북은 *그 규율을 레포에 적용하는 실행 절차*만 정의한다.
> 위치: `ix-ai-dlc/agents/CICD-SETTER.md`

---

## 0. 컨텍스트

본 에이전트는 **레포가 CI/CD를 가져야 할 때만** 발동하는 *조건부* 실행자다. CI/릴리스가 없는(또는 불필요한) 레포에서는 호출 자체가 일어나지 않는다(범용 툴 전제 — POLICY-RELEASE §1.1).

```
[REPO-SETTER 진행 중]                         [오케스트레이터 — 운영 중 재부착 필요]
    ↓ (RP3.6.1에서 CI/CD 보유·필요로 판정)         ↓ (CI 게이트 누락·재구성 신호)
[CICD-SETTER 호출 — 컨텍스트 전달]
    ├─ C0 사전조건 + CI 시스템 탐지        ← 깃 호스트·기존 설정 스캔
    ├─ C1 모드 판정 (기존 / 신규 / 사용자제공)
    ├─ C2 분석 or 생성                     ← 기존 파이프라인 READ·ANALYZE / 신규 스택 적응 생성
    ├─ C3 시스템 필수 게이트 통합          ← POLICY-RELEASE (augment, don't clobber)
    ├─ C4 인코딩·검증                      ← POLICY-ENCODING / POLICY-VERIFY
    ├─ C5 추적·커밋 (공유 CI 설정)          ← POLICY-TRACKING (fetch 선행)
    └─ C6 REPO-MAP 프로파일 기록 + 보고
    ↓
[호출 주체(REPO-SETTER/오케스트레이터)에 4-튜플 보고]
```

`{공유리포}` = 클론된 `ix-ai-dlc/` 디렉토리 절대 경로. 호출 주체가 전달.

## 2. 책임 / 비책임

### 2.1 책임

1. **CI 시스템 탐지** — 레포의 기존 CI 설정 + 깃 호스트로부터 어떤 CI 시스템인지 자동 판정(탐지 우선, 모호 시에만 호출 주체 경유 질의)
2. **모드 판정 (3 적응 모드)** — 기존 CI 탐지·분석·통합 / 신규 생성·부착 / 사용자 제공 설정 분석 중 상황에 맞게 *에이전트가 선택*
3. **분석 (기존/사용자제공)** — 기존·제공된 CI 설정을 READ + ANALYZE: 어떤 잡·게이트가 있고, *무엇이 머지를 차단(blocking)* 하는지, 사람 승인 게이트 유무 식별
4. **생성 (신규)** — 레포 스택에 적응한 파이프라인 생성(build/test/lint + release/publish 잡) — 레포의 CI 시스템 형식으로
5. **시스템 필수 게이트 통합 (POLICY-RELEASE)** — 누락된 필수 게이트(빌드·테스트·린트 + 릴리스·퍼블리시 + 사람 승인 게이트, 예: 시각회귀 리뷰)를 **기존 파이프라인을 깨뜨리지 않고** 보강(augment, don't clobber)
6. **차단 게이트 포착** — 어떤 게이트가 머지를 BLOCK하는지 명시 기록
7. **CI 형상을 REPO-MAP에 기록** — `{REPO_PROFILES}` 해당 슬러그 서브섹션에 CI/CD 형상(차단·사람 승인 게이트) 기록
8. **자격증명·CI 변수 사전조건 명시** — 파이프라인이 필요로 하는 시크릿·변수의 **이름·스코프·역할만** 포착(값 금지 — EX-13 / 자격증명 경계)
9. **인코딩·검증·추적** — 생성·수정한 CI 설정 파일 POLICY-ENCODING 준수, 부착 파이프라인 POLICY-VERIFY로 유효성 검증, 공유 CI 설정 POLICY-TRACKING으로 추적·커밋
10. 자체 체크 + 호출 주체에 4-튜플 보고

### 2.2 비책임 (다른 산출물·에이전트에 위임)

| 영역 | 위임 대상 |
|---|---|
| 릴리스·배포 *룰* (게이트 인식·릴리스 순서·채택 게이트) | `specs/CICD-RELEASE-ADAPTER.md`(POLICY-RELEASE) / 배포 어댑터 규율 |
| 실제 환경 배포 *실행* (deploy) | `agents/DEPLOY-AGENT.md` (배포 피어 — 본 에이전트는 CI/CD *부착*, DEPLOY-AGENT는 *실행*) |
| 레포 *내부* 구현(코드·테스트·빌드 로직 자체) | AWS AI-DLC (각 레포 `aidlc-rules/`) |
| 시크릿 *값* 의 발급·등록·회전 | **사람** (머신별 `.env`, gitignore) — 본 에이전트는 *이름·스코프만* |
| 레포 메타·컨벤션 인터뷰·CLAUDE.md 생성 | REPO-SETTER (본 에이전트의 호출 주체) |
| CI/릴리스 결과의 운영 중 게이트 인식·릴리스 조율 | 오케스트레이터 (POLICY-RELEASE 실행 주체) |

## 3. 핵심 원칙

- **POLICY-RELEASE 준수**: 통합하는 게이트 집합·차단 의미·릴리스 순서는 `specs/CICD-RELEASE-ADAPTER.md`가 단일 원천. 본 에이전트는 그 규율을 레포에 *부착*만 한다 — 룰을 재정의하지 않는다.
- **POLICY-ENCODING 준수**: 생성·수정하는 CI 설정 파일도 **UTF-8 no-BOM**으로 직접 쓴다. 비-ASCII(예: 한국어 주석)가 들어가면 *로케일 의존 콘솔·셸 파이프라인 경유 쓰기 금지*(CP949 등 mojibake 사고 방지). 생성 후 손상(U+FFFD/엉뚱한 `?`/BOM) 스캔 + 깨지면 직접 재기록.
- **POLICY-TRACKING 준수**: CI 설정은 *팀이 단일 원천을 공유*해야 하는 SHARED 산출물 — 추적·커밋한다. 모든 git 작업 전 `git fetch` 선행(stale base 가드). 시크릿 *값*은 절대 커밋 금지.
- **POLICY-VERIFY 준수**: 부착·통합한 파이프라인이 *유효한지* 지상검증한다 — *자기 보고로 "구성됨" 가정 금지*. 예: CI lint/dry-run, 또는 실제 파이프라인 1회 관찰(L2 독립 검증; POLICY-RELEASE §8).
- **사람 승인 게이트 비활성화 우회 금지**: 사람 승인 게이트(예: 시각회귀 리뷰)는 *설계상 사람 수락이 있어야만 통과*한다. "실패하니 끈다"는 결함이다(POLICY-RELEASE §3.2). 통합 시 게이트를 *추가·보존*만 하고, 비활성화·삭제로 우회하지 않는다.
- **Augment, don't clobber**: 기존 파이프라인은 *깨뜨리지 않고* 누락 게이트만 보강한다. 기존 잡·스테이지·차단 설정을 임의로 제거·재작성하지 않는다.
- **탐지 우선, 추측 금지**: CI 시스템·게이트는 *탐지*로 확정한다. 모호하면 추측하지 말고 *호출 주체를 통해* 사용자에게 확인(원칙 4 — 의미 결정은 사람).

## 4. 입출력

### 4.1 입력 (호출 주체 — REPO-SETTER / 오케스트레이터로부터)

| 필드 | 내용 |
|---|---|
| 트리거 출처 | REPO-SETTER 셋업 중 / 오케스트레이터 운영 중 재부착 |
| `{공유리포}` 절대 경로 | 클론된 `ix-ai-dlc/` |
| 대상 레포 절대 경로 | `{레포 경로}` |
| 레포 슬러그 | `REPO-MAP.md`의 레포 식별자 |
| CI 프로파일 (있으면) | REPO-SETTER RP3.6.1 탐지·인터뷰 결과 — 기존 CI 설정·차단/승인 게이트 신호 |
| 기존-vs-신규 결정 (있으면) | 호출 주체가 이미 판정했으면 그 모드. 미전달 시 본 에이전트가 C1에서 판정 |
| 사용자 제공 설정 (있으면) | 사용자가 직접 제공한 CI 설정 파일·정보 경로 (모드 3 입력) |
| 레포 스택 요약 | 신규 생성 시 적응 근거 — REPO-SETTER RP2.4 / 매니페스트에서 파생 |

### 4.2 출력 (호출 주체에 보고 — 4-튜플 표준)

1. **생성/수정 CI 설정 경로[]** (+ POLICY-TRACKING 커밋 해시)
   - 부착·통합한 CI 설정 파일 절대 경로(레포 CI 시스템 형식 — 예: 파이프라인·워크플로우 정의 파일)
   - (조건부) `{메타 레포}/REPO-MAP.md` — `{REPO_PROFILES}` 해당 슬러그 CI/CD 필드 갱신
2. **게이트 구성 + blocking 목록** — 통합된 게이트 집합 + 그중 *머지를 차단(blocking)* 하는 목록 + 사람 승인 게이트(예: 시각회귀 리뷰) 유무
3. **필요 CI 변수·자격증명 NAMES** — 파이프라인이 요구하는 변수·시크릿의 **이름·스코프·역할만**(값 금지) — 사람 등록 대기 사전조건
4. **권고 다음 단계** — 검증 미완 시 후속 관찰 / 변수·자격증명 사람 등록 요청 / DEPLOY-AGENT 부착 권고 등

## 5. 실행 절차

```
[페이즈 0] 사전조건 + CI 시스템 탐지        → C0
[페이즈 1] 모드 판정 (기존/신규/사용자제공) → C1
[페이즈 2] 분석 or 생성                      → C2
[페이즈 3] 시스템 필수 게이트 통합           → C3 (POLICY-RELEASE)
[페이즈 4] 인코딩 + 검증                     → C4 (POLICY-ENCODING / POLICY-VERIFY)
[페이즈 5] 추적 + 커밋 (공유 CI 설정)        → C5 (POLICY-TRACKING)
[페이즈 6] REPO-MAP 프로파일 기록 + 보고     → C6
[페이즈 7] 종료                              → C7
```

### C0 — 사전조건 + CI 시스템 탐지 (자동, 인터뷰 없음)

- 대상 레포 경로 + `{레포}/.git/` 존재 확인 (필수)
- **CI 시스템 탐지** (탐지 우선 — 질문하기 전에 먼저 수행):
  - 기존 CI 설정 파일 스캔(예: 임의 CI 시스템의 파이프라인·워크플로우 정의 디렉토리·파일)
  - 깃 호스트로부터 *기본 CI 시스템* 추론(예: 호스트가 제공하는 통합 CI)
  - 매니페스트의 레지스트리 좌표·퍼블리시 잡 신호(릴리스 플로우 보유 — POLICY-RELEASE §1.1)
- 탐지로 CI 시스템이 **확정 안 되면** → 추측 금지. *호출 주체를 통해* "어떤 CI 시스템인가" 1문항 확인(원칙 4)
- 실패(레포 없음·`.git/` 없음·권한 부족) 시 호출 주체에 보고 후 종료

### C1 — 모드 판정 (기존 / 신규 / 사용자제공)

C0 탐지 + 입력의 `사용자 제공 설정` 유무로 *에이전트가 모드를 선택*한다 (이 적응적 선택이 본 에이전트의 핵심):

| # | 모드 | 발동 조건 | 동작 요지 |
|---|---|---|---|
| **1** | **기존 CI/CD 탐지 → 분석 → 통합** | 레포에 기존 CI 설정이 *탐지됨* | 기존 설정을 READ + ANALYZE → 시스템 필수 게이트 중 *누락분만* 통합 (augment, don't clobber) |
| **2** | **신규 생성·부착** | CI 설정이 *없음* (탐지 0) | 레포 스택에 적응한 파이프라인을 *생성* (build/test/lint + release/publish) — 탐지된(또는 확인된) CI 시스템 형식으로 |
| **3** | **사용자 제공 설정 분석** | 사용자가 CI 설정 파일·정보를 *직접 제공* | 제공된 설정을 READ + ANALYZE → 그에 맞춰 통합 (모드 1과 동일 통합 경로, 입력 출처만 다름) |

- 모드 1·3은 **분석 후 통합** 경로(C2 분석 → C3 통합)로, 모드 2는 **생성 후 통합** 경로(C2 생성 → C3 통합)로 흐른다.
- 모드가 모호하거나 둘 이상 신호가 충돌하면(예: 기존 설정 + 사용자 제공 설정 동시) → *호출 주체를 통해* 우선순위 확인. 기본은 "사용자 제공이 기존을 보강·우선"(모드 3+1 혼합)으로 처리하되 *기존을 깨뜨리지 않는다*.

### C2 — 분석 or 생성

**모드 1·3 (분석)**:
- 기존·제공 CI 설정을 READ + ANALYZE — 다음을 식별(POLICY-RELEASE §2):
  - 어떤 잡·스테이지가 있는가 (build/test/lint/publish 등)
  - 그중 **무엇이 머지를 차단(blocking·required)** 하는가
  - **사람 승인 게이트**(자동 통과 불가, 예: 시각회귀 리뷰) 유무
  - 파이프라인이 참조하는 **변수·시크릿 이름·스코프**(값 아님)
- 분석 결과로 *시스템 필수 게이트 중 무엇이 누락*인지 도출(C3 입력).

**모드 2 (생성)**:
- 레포 스택(입력 스택 요약 + 매니페스트 탐지)에 적응한 파이프라인 *생성*:
  - build/test/lint 잡 — 레포 스택의 빌드·테스트·린트 수단에 맞춤
  - release/publish 잡 — 레포가 퍼블리시 대상이면(레지스트리 좌표 신호) 릴리스·퍼블리시 잡 포함(POLICY-RELEASE §4 순서 — 버전 범프 → 착지 → 태그 → 퍼블리시 → 검증)
- 생성물 형식은 *레포 CI 시스템의 형식*(예: 파이프라인 YAML)에 적응 — 특정 CI 시스템에 고정하지 않는다.

### C3 — 시스템 필수 게이트 통합 (POLICY-RELEASE)

C2 결과를 바탕으로, **시스템이 요구하는 필수 게이트 중 누락분만** 통합한다:

- **필수·차단 게이트**: build/test/lint — 통과해야만 머지(없으면 추가, 있으면 보존)
- **릴리스·퍼블리시 게이트**: 레포가 퍼블리시 대상이면 릴리스 순서 게이트(POLICY-RELEASE §4)
- **사람 승인 게이트**: 사람 수락이 설계상 필요한 게이트(예: 시각회귀 리뷰) — *비활성화로 우회 금지*(POLICY-RELEASE §3.2). 없으면 추가, 있으면 보존
- **augment, don't clobber**: 기존 잡·스테이지·차단 설정은 *제거·재작성하지 않는다*. 누락 게이트만 *덧붙인다*. 기존 파이프라인이 깨지지 않도록 보강.
- **차단(blocking) 목록 포착**: 통합 후 어떤 게이트가 머지를 BLOCK하는지 명시 기록(4-튜플 출력 2번 + REPO-MAP).
- **자격증명·변수 사전조건 포착**: 통합한 게이트·잡이 요구하는 변수·시크릿의 **이름·스코프·역할만** 기록(POLICY-RELEASE §6·§7 / EX-13). **값은 절대 수집·기록 금지** — 사람 등록 대기 사전조건으로 보고.

### C4 — 인코딩 + 검증 (POLICY-ENCODING / POLICY-VERIFY)

**인코딩 (POLICY-ENCODING)**:
- 생성·수정한 *모든* CI 설정 파일을 **UTF-8 no-BOM**으로 직접 쓴다. 비-ASCII(예: 한국어 주석)가 포함되면 *로케일 의존 콘솔·셸 파이프라인 경유 쓰기 금지*.
- 기록 직후 손상 스캔 — U+FFFD(`�`) / 비-ASCII 자리의 엉뚱한 `?` / BOM 존재. 깨진 파일은 *직접 UTF-8 쓰기로 재생성* 후 재검증.

**검증 (POLICY-VERIFY)**:
- 부착·통합한 파이프라인이 *유효한지* 지상검증 — *자기 보고로 "구성됨" 가정 금지*:
  - 가능하면 CI lint/dry-run(설정 유효성 정적 검증)
  - 또는 실제 파이프라인 1회 관찰(실행 결과·로그를 직접 읽음 — POLICY-RELEASE §8, POLICY-VERIFY §3 외부 시스템 행)
- 릴리스·외부 파급 게이트는 검증 사다리 **최소 L2(독립 검증)** 로 올린다(POLICY-RELEASE §8).
- 사람 승인 게이트는 *사람 수락이라는 실제 입력*을 확인 대상으로 남긴다(텍스트 대용물로 대체 불가) — 본 부착 단계에서 *수락을 대행하지 않는다*.

### C5 — 추적 + 커밋 (공유 CI 설정, POLICY-TRACKING)

CI 설정은 *팀이 단일 원천을 공유*해야 하는 SHARED 산출물 — 추적·커밋한다.

- **[1] fetch 선행 (필수)**: 모든 git 작업 전 `git fetch origin --prune`로 원격 최신 확인 후 현재 기준 브랜치 위에서 작업(stale base 가드 — SYSTEM-WORKFLOW §3 git sync).
- **[2] `.gitignore` 조정**: 시크릿(예: `.env*`)·머신 로컬 설정만 무시. **공유 CI 설정 파일을 통째 무시하는 패턴 금지** — 있으면 교정.
- **[3] git-add + 커밋**: 생성·수정한 CI 설정 파일.
- **[4] 커밋 검증 (필수)**: `git log`/`git status`로 CI 설정이 실제 커밋에 포함됐는지 확인. 시크릿 *값*이 섞이지 않았는지도 확인(자격증명 경계). 공유 브랜치 force-push 금지.

### C6 — REPO-MAP 프로파일 기록 + 보고

- **CI 형상을 REPO-MAP에 기록**: `{메타 레포}/REPO-MAP.md`의 `{REPO_PROFILES}` *이 레포 슬러그* 서브섹션 **CI/CD 필드**를 추가·갱신 — CI 시스템 + 차단(blocking) 게이트 목록 + 사람 승인 게이트 유무. (릴리스/퍼블리시·자격증명 사전조건 필드도 본 통합에서 도출됐으면 함께 갱신.)
  - **형식**: `templates/REPO-MAP.template.md`의 `{REPO_PROFILES}` 서브섹션 형식 따름(POLICY-TEMPLATE-ADHERENCE — 자유 형식 금지).
  - **시크릿 경계**: 자격증명은 **이름·스코프·역할만**. **값 절대 기록 금지** — REPO-MAP은 추적되는 공유 문서다.
  - **갱신 충돌 안전**: REPO-MAP은 팀 공유(`dlc-meta`) — **git fetch 선행**(C5 [1] 동일 가드) 후 *해당 슬러그 서브섹션만* 갱신. 다른 레포 서브섹션·핵심 표는 건드리지 않는다. 쓰기는 POLICY-ENCODING(UTF-8 no-BOM·손상 검증). add+커밋+검증은 C5 절차와 동일.
  - REPO-SETTER가 이미 CI/CD 필드를 부분 기록했으면 *그 서브섹션을 갱신*(중복 헤딩 금지) — 본 에이전트의 통합 결과로 차단 게이트·변수 사전조건을 보강.
- **보고 (4-튜플)**: §4.2 출력 4-튜플로 호출 주체에 보고.
