# ERROR-POLICY.md — 에러·예외 처리 정책

> ORCHESTRATOR-AGENT.md §6의 본문.
> 책임 아닌 *횡단 정책* — 모든 단계에서 적용 (D-Stage2-9, c-3 선택).
> 위치: `ix-ai-dlc/agents/orchestrator/ERROR-POLICY.md`
> 로딩 시점: 예외 감지 시
> **이 파일은 공유 룰북의 *디폴트* (전 프로젝트 공통, 불가침).** 프로젝트별 (EX, 레포) 특화는 DP-9(신호 ③)이 `dlc-meta/ORCHESTRATOR.md` *예외 오버라이드*에 쌓음 — 디폴트 위에 얹히며, 진화가 이 공유 파일을 건드리지 않음 (D-Stage3-6).

## 1. 처리 정책 카테고리 (5)

| 코드 | 정책 | 의미 | 사용 빈도 |
|---|---|---|---|
| **A. RETRY** | 자동 재시도 | transient 에러 (네트워크 일시 단절, 일시 락 등). **1회 자동** → 실패 시 다른 정책으로 escalate | 낮음 |
| **B. ASK** | 사용자 컨펌 + 결정 | 가장 일반적. 멀티초이스로 옵션 제시 | 높음 |
| **C. FALLBACK** | 폴백 (대체 경로) | AWS 실패 시 우리 단순 모드, 한 도구 실패 시 다른 도구 등 | 중간 |
| **D. PAUSE** | 그레이스풀 중단 + 상태 저장 | 복구 가능. 사이클 ID 기반 재개. 사용자 명시 취소·컨텍스트 한도 등 | 중간 |
| **E. ABORT** | 즉시 중단 + 에러 보고 | 복구 불가. 사이클 종료. 정합성 실패·룰셋 mismatch 등 | 낮음 |

### 1.1 정책 escalation 흐름

```
A RETRY (1회) → 실패 → B ASK (사용자 컨펌)
                              ↓
                       사용자 결정에 따라
                              ↓
                  C FALLBACK / D PAUSE / E ABORT
```

A는 *항상 1회만*. 무한 재시도 X.

| 항목 | 내용 |

| 후속 액션 | 검증 통과 → [6] 응답 검토 진입 / 재차 실패 → 영향 산출물 재작업(해당 EX로 회귀) |
| 라벨 | `specs/VERIFICATION.md` **POLICY-VERIFY** 인용. POLICY-ENCODING "생성 후 검증" 도 본 EX의 한 사례 |

### EX-13 — 권한·스코프 실패 (forbidden / insufficient-scope)

| 항목 | 내용 |
|---|---|
| 감지 | 권한·자격증명이 필요한 작업이 *forbidden / insufficient-scope / unauthorized* 류 에러 반환 (예: API가 권한 부족 응답 — `read_api`만 있고 `api` 없음 등은 *예시일 뿐*) |
| 정책 | **B ASK / D PAUSE** — 사람이 올바른 스코프의 자격증명을 제공하도록 요청·대기. 토큰 발급·회전은 *사람 게이트*(자격증명 경계 원칙, §3 공통 정책). |
| 사용자 통지 | 항상 |
| 진단 절차(일반) | 1) *실제 에러 본문* 을 읽는다(추정 금지) 2) 토큰의 **신원(whoami)** 확인 3) 토큰의 **스코프** 확인 4) 해당 주체의 대상 리소스에 대한 **역할·멤버십** 확인 5) **스코프 부족 vs 역할 부족 vs 토큰 종류 오류** 를 구분 6) *맞는 층위* 에서 교정 |
| 복구 시나리오 | 진단으로 부족 층위 특정 → 사람에게 *올바르게 스코프된 자격증명* 발급·회전 요청 → 제공 후 재시도. 시크릿 값은 `.env` 에서 읽기만, 추적 파일에 커밋 금지(§3). |
| 후속 액션 | 자격증명 갱신 후 원래 작업 재진입 / 미제공 시 D PAUSE (사이클 ID 저장) |

### EX-14 — 배포·롤백 실패 (deploy / rollback failure)

| 항목 | 내용 |
|---|---|
| 감지 | 배포(deploy)가 실패 / 배포 후 헬스·스모크 검증(health·smoke)이 실패 / 롤백(rollback) 자체가 실패 — 셋 중 하나 |
| 정책 | **배포·헬스 실패 → 직전 정상 상태로 자동 롤백(C FALLBACK)** 후 재검증. **비프로덕션(non-prod)** 은 롤백을 *자율 진행* 가능. **프로덕션(production)** 은 서비스 복원을 위해 롤백을 *안전 디폴트* 로 진행하되 **사람에게 에스컬레이트(D PAUSE)**. **롤백 자체가 실패하면 → D PAUSE + 사람 에스컬레이트** (환경을 *알 수 없는 상태* 로 방치 금지 — POLICY-VERIFY대로 *정직하게 보고*) |
| 사용자 통지 | 항상 (프로덕션 배포·헬스·롤백 실패 시), 비프로덕션 자율 롤백 시 결과 보고 |
| 복구 시나리오 | 1) 배포·헬스 실패 감지 → 직전 정상 상태로 자동 롤백(C FALLBACK) → 롤백 후 헬스 재검증(지상검증, POLICY-VERIFY) 2) 비프로덕션은 롤백·재검증 자율 진행 3) 프로덕션은 롤백으로 서비스 복원 후 사람 에스컬레이트(D PAUSE) 4) 롤백 자체 실패 시 환경 상태를 *정직하게 보고* 하고 사람 에스컬레이트(D PAUSE) |
| 후속 액션 | 롤백·재검증 통과 → 원인 진단 후 배포 재시도 (A-4 컨펌 재경유) / 롤백 실패 → D PAUSE (사이클 ID 저장) |
| 라벨 | `specs/DEPLOY-ADAPTER.md` **POLICY-DEPLOY** 인용 — 실행 주체는 `agents/DEPLOY-AGENT.md`. 프로덕션 배포는 *상류에서 사람 게이트* (DECISIONS A-4)이므로 본 EX는 주로 *실행·헬스·롤백* 단계를 다룬다. EX-13(자격증명)과 구분 — 본 EX는 배포 *실행·검증* 실패 |

## 3. 기록·추적 (audit.md 호환)

분기점 기록(DECISIONS.md §3)과 *동일 형식* + 예외 필드.

### 3.1 예외 기록 형식

```
[YYYY-MM-DD HH:MM:SS] EX-N: {예외 이름}
  Detected: {감지 입력}
  Policy: {적용 정책 (A~E) + 이유}
  Result: {성공 / 실패 / escalate}
  Confirm: {사용자 컨펌 응답 (해당 시)}
  Next: {후속 액션}
  (관련 분기점: DP-N — 해당 시)
```

### 3.2 기록 위치

```
dlc-meta/cycles/{cycle-id}/audit.md
```

분기점 기록과 동일 파일. append-only.

### 3.3 DP-9 데이터 소스 흡수

예외 기록도 DP-9(진화 능동 제안) 입력. 반복 예외 패턴이 룰북 갱신 트리거 후보:
- 동일 EX-N 반복 → 휴리스틱 갱신 또는 정책 변경 제안
- 동일 escalate 경로 → 디폴트 정책 변경 제안
- 동일 사용자 컨펌 응답 → 자율화 가능성 제안

## 4. EX-9 핸드오프 자동 생성 본격 명세

> 메타적: 우리가 지금 수동으로 만드는 `HANDOFF_v{n}.md` 작업이 시스템 운영 중 자동화됨. dogfooding.

### 4.1 트리거 카탈로그

| 트리거 | 조건 | 정책 |
|---|---|---|
| **컨텍스트 한도 임박** | 토큰 사용량 ≥ 80% 또는 한도 근접 신호 | 자동 (사용자 통지) |
| **사용자 명시 요청** | "핸드오프 만들어줘" / "다음 세션용 정리" 등 | 즉시 |
| **사이클 종료 시 (옵션)** | 사이클 종료 + 다음 사이클 컨텍스트 인계 필요 | 사용자 컨펌 |
| **수동 안전 백업** | 큰 분기점 결정 직후 (예: D-Stage2-7 같은) | 사용자 컨펌 |

### 4.2 채움 주체 — HANDOFF-WRITER 서브 에이전트

- 원칙 1(호출 시점·책임 다르면 분리) 정합 — 핸드오프 작성은 *드물고 큰 컨텍스트 작업*
- 위치: `ix-ai-dlc/agents/HANDOFF-WRITER.md` (룰북)
- 호출 방식: 오케스트레이터가 EX-9 트리거 감지 시 호출
- 입력: 현재 사이클 상태 + audit.md 누적 + 트리거 사유
- 반환: 4-튜플 (표준 — `(생성 파일 경로[], 정합성 체크, 인터뷰 응답 원본, 권고 다음 단계)`)

### 4.3 자동 생성 포맷

`ix-ai-dlc/templates/HANDOFF.template.md` 변수 구조 (별도 산출물).

필수 섹션 (현재 정의):
- 메타 (버전, 작성일, 트리거 사유, 응답 언어)
- 프로젝트 정체성
- 현재 진행 상황 (Stage별 ✓/진행/대기)
- ★ 직전 핵심 결정 (신규·중요 D-* 강조)
- 최근 Decision Log (전체 또는 마지막 N개)
- 작업 흐름 현재 위치 (단계, 컨펌 대기 사항)
- 큐 (잔여 작업·신규 발생 항목)
- 다음 작업 가이드 + 새 세션 첫 한마디 권장
- 필수 첨부 파일 목록

### 4.4 산출 위치

```
dlc-meta/cycles/{cycle-id}/handoff-v{n}.md
```

- cycles/에 영구 저장
- 사용자에게 다운로드 형태로 출력 (예: `present_files`)
- 같은 사이클 내 다중 핸드오프 가능 (v{n} 증가)

### 4.5 DP-9 데이터 소스

핸드오프 시점·빈도 패턴이 누적되면 *컨텍스트 분할 전략* 자체가 진화 대상:
- 너무 자주 EX-9 발생 → 인터페이스 디테일 더 분리 제안
- 특정 시점에서 항상 트리거 → 구조 변경 제안
- 사용자가 사이클 종료 시 옵션을 자주 거부 → 트리거 정책 조정

## 5. AWS ↔ 우리 예외 전파

### 5.1 AWS → 우리

- AWS `aidlc-state.md` + `audit.md` 모니터링
- AWS 실패 entry 발견 → EX-2 (호출 실패) 또는 EX-7 (실행 실패) 트리거
- 흡수: 책임 7 (사이클 로그 집계) — 우리 audit.md에 AWS audit entry 매핑 통합

### 5.2 우리 → AWS

- 트리거 취소·재실행 명령 — `AWS-ADAPTER.md` 인터페이스 통해
- `execution-plan.md` 갱신 후 재트리거

### 5.3 양방향 audit 통합

- 우리 `dlc-meta/cycles/{cycle-id}/audit.md` (시스템 단위)
- AWS `{레포}/aidlc-docs/audit.md` (레포 단위)
- 시스템 단위 audit이 *레포 단위 audit을 흡수 집계* (책임 7)
- 형식 호환은 `CYCLE-LOG.md` 명세 위임

### 5.4 본격 인터페이스 명세

**`AWS-ADAPTER.md` 위임**. 본 파일은 *정책*만, 실제 호출 메커니즘은 어댑터.

## 6. 복구 메커니즘

### 6.1 사이클 ID 기반 재개

- 사이클 로그에 마지막 결정 지점 + 컨텍스트 스냅샷 저장
- 새 세션에서 사이클 ID 던지면 재개 가능
- 형식 명세는 `CYCLE-LOG.md` 위임

### 6.2 핸드오프 문서 자동 (EX-9)

§4 본격 명세 참조.

### 6.3 AWS partial completion

- `execution-plan.md`의 일부 STEP만 완료된 상태에서 재개 가능 (AWS 표준 지원)
- AWS `aidlc-state.md`가 STEP별 상태 추적 — 우리는 그것 보고 어디서부터 재개할지 결정
