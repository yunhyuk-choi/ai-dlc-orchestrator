## 1. 종료 단위 (3)

| 단위 | 의미 | 빈도 | 처리 위치 |
|---|---|---|---|
| **사이클 종료** | 한 인텐트의 한 작업 사이클 마무리. 다음 인텐트 받을 준비. | 빈번 | §2~4 |
| **세션 종료** | 한 채팅 세션 끝. 컨텍스트 단절. 핸드오프로 다음 세션 인계. | 중간 | §5 |
| **운영 모드 종료** | 시스템 자체 종료. 사용자가 더 이상 사용 안 함. | 드뭄 | §6 |

세 단위는 *독립*. 동시 발생 가능 (예: 사이클 종료 + 세션 종료 동시).

## 2. 정상 사이클 종료 신호 (4)

| 신호 | 출처 | 자동 vs 컨펌 |
|---|---|---|
| 인텐트 완료 + 사용자 명시 신호 ("다 됐어", "마무리") | 사용자 | **자동** (재확인 1회) |
| STEP 10(문서화) 완료 + 사용자 검토 통과 | 워크플로우 | 사용자 컨펌 |
| AWS `execution-plan.md` 전체 STEP 완료 + cross-repo 검증 통과 | 통합 신호 | 사용자 컨펌 |
| 분석·조사 인텐트 — STEP 1~2 완료 후 결과 보고만 | 워크플로우 | **자동** |

### 2.1 자동 vs 컨펌 분기

- **자동**: 명시적 신호 또는 인텐트 유형상 보고만 필요한 경우 (분석·조사)
- **컨펌**: 큰 산출물·다중 레포 변경 후 — 사용자 검토 게이트 필수

### 2.2 재확인 1회 정책

사용자 명시 신호("끝내자")라도 *재확인 1회* 진행:
- "사이클 종료할게요. 추가로 할 일 없나요?"
- 추가 작업 신호 시 → 사이클 종료 보류 + 추가 인텐트 분류 (DP-1 재진입)
- 없음 확인 시 → §4 종료 후처리 진입

## 3. 예외 종료 신호

ERROR-POLICY.md 정책 결과로 사이클 종료:

| 신호 | 정책 | 결과 |
|---|---|---|
| EX-5 사용자 취소 | D PAUSE | 사이클 로그 저장 후 종료. 재개 가능 (사이클 ID 기준) |
| EX-3 AWS 룰셋 mismatch | E ABORT | 복구 불가. CYCLE-CLOSER 호출 없이 즉시 중단 가능 |
| EX-6 정합성 체크 실패 (수정 거부 시) | E ABORT | 동일 |
| 기타 escalate된 예외 (사용자가 ABORT 선택) | E ABORT | 동일 |

### 3.1 PAUSE vs ABORT 분기

- **D PAUSE** → CYCLE-CLOSER 호출 (정상 후처리). 재개 가능.
- **E ABORT** → CYCLE-CLOSER 호출 가능하나 *간소화 모드* (정상 후처리 일부 스킵). 또는 즉시 중단.

ABORT의 간소화 모드 결정 기준:
- 사이클 진행률 < 30% → 즉시 중단 (후처리 의미 없음)
- 사이클 진행률 ≥ 30% → CYCLE-CLOSER 간소화 호출 (부분 산출물 보존)

기준값(30%)은 휴리스틱 — 운영 중 EVOLUTION(DP-9)으로 조정 가능.

## 4. 종료 후처리 — CYCLE-CLOSER 호출

```
[종료 신호 감지]
       ↓
[종료 사유 분류 — 정상 / 예외]
       ↓
[CYCLE-CLOSER 호출 — 컨텍스트 전달]
       ↓
[CYCLE-CLOSER 결과 수신 — 4-튜플]
       ↓
[회고(RETROSPECTIVE) 캡처 — §4.4]   ← 종료 직전/시점, EVOLUTION 입력으로 라우팅
       ↓
[사용자 보고 — 사이클 요약]
       ↓
[다음 사이클 대기 모드]
```

### 4.1 CYCLE-CLOSER 호출 컨텍스트 (4 필드)

| 필드 | 내용 | 단일 원천 |
|---|---|---|
| **사이클 ID** | 종료할 사이클 식별자 | `dlc-meta/cycles/{cycle-id}/` 디렉토리명 |
| **종료 사유** | 정상 / 예외 분류 + 구체 사유 | DP-8 결정 기록 또는 EX-N |
| **영향 레포·산출물 목록** | 사이클 중 영향 받은 레포 + 생성·수정 파일 | audit.md 누적 결정 추적 |
| **audit.md 경로** | 결정 추적 데이터 원천 | `dlc-meta/cycles/{cycle-id}/audit.md` |

### 4.2 확장 게이트

**CYCLE-CLOSER.md 본문 작성 시 필요 식별되면 역방향으로 인터페이스 확장 허용.**

가능한 확장 후보 (큐):
- AWS audit.md 경로 (레포별, 영향 레포에서 파생 가능)
- execution-plan.md 경로 (AWS 통합 추적)
- 사이클 중 DP-9 진화 제안 누적
- 사용자 원본 인텐트 (audit.md 첫 entry에서 파생 가능)

→ 원칙 6(단일 원천 = audit.md)에 의해 위 모두 *파생 가능*. **현재 4 필드 유지**. CYCLE-CLOSER.md 작성 시 필요 확인되면 큐 항목 등재.

### 4.3 CYCLE-CLOSER 반환 (4-튜플 표준)

서브 호출 표준 반환:
1. 생성 파일 경로[] — 후처리 산출물 (티켓 / 커밋 / 보고서)
2. 정합성 체크 결과 — 사이클 산출물 정합성
3. 인터뷰 응답 원본 — 사용자 컨펌 응답 (재확인 등)
4. 권고 다음 단계 — 다음 사이클 권장 또는 대기

### 4.4 회고(RETROSPECTIVE) — 종료 절차의 정식 단계 (→ EVOLUTION 입력)

사이클 종료 시점(또는 직전)에 **이번 사이클에서 무엇이 어긋났나 / 룰북이 무엇을 놓쳤나 / 무엇을 (다시) 발견했나**를 *구조화 입력*으로 캡처하고, 이를 **진화 메커니즘(EVOLUTION / DP-9)**으로 라우팅한다. 종료 절차의 *정의된 일부* — 빠뜨리지 않는다.

**왜 정식 단계인가**: 기존 DP-9 진화 입력은 *런타임 행동 로그*(index.md 롤업 — ②①③④ 신호)에 한정됐다. 그러나 "룰북에 X가 없어서 헤맸다", "환경 gotcha를 또 만났다" 같은 *서술적 교훈*은 로그 카운팅으로 안 잡힌다. 회고는 이 빈틈을 메우는 **사람·에이전트 발 1급 신호** — 사람이 수동으로 회고를 *촉발*하지 않아도 매 사이클 종료가 자동으로 캡처한다.

**입력 소스 (이번 사이클 audit.md 원천)**:
- 본문 `EX-*` 예외 entry — 무엇이 실패/재시도됐나
- `CORRECTION` / `SCOPE-EXPANSION` / `CYCLE-REOPEN` entry (CYCLE-LOG §9.5·§10) — 잘못된 가정·범위 오판·재오픈
- 정합성 체크 미스(4-튜플 ②) — 검증이 놓친 것
- CYCLE-CLOSER가 마주친 환경 gotcha·반복 마찰

**입도 — 사이클 성격에 따라 (경량 ↔ 충실)**:
- **사소·정형 사이클** (예외 0, 범위 안정, 기존 라우팅 그대로): *경량* — 회고 한두 슬롯만 채우거나 "특이사항 없음"으로 1줄 마감. 강제 장문 금지(인지 과부하 방지).
- **충실·신규 사이클** (예외 발생·sprawl·재오픈·새 인텐트 유형·룰북 갭 체감): *풀 회고* — `templates/RETROSPECTIVE.template.md` 전 슬롯 채움. 특히 *룰북 갭*과 *제안 변경*을 명시.

**산출·라우팅**:
- 산출 위치: `dlc-meta/cycles/{cycle-id}/retrospective.md` — `templates/RETROSPECTIVE.template.md` 변수 채움.
- audit.md에 종료 직전 `PROGRESS: RETROSPECTIVE captured` 1줄 append (append-live, CYCLE-LOG §5.1).
- **EVOLUTION 라우팅**: 회고의 *제안 룰북 변경* 슬롯이 **DP-9 진화 휴리스틱의 1급 입력원**이 된다 (`specs/EVOLUTION.md` §2·§3.E). 로그 기반 ②①③④ 신호와 *나란히* 진화 제안을 형성. 적용 게이트·경계는 EVOLUTION 본문이 그대로 관할 (항상 컨펌, dlc-meta 인스턴스 한정).

> 경계: 회고는 *캡처·라우팅*만 한다. 룰북을 *바꾸는* 결정·적용은 DP-9(EVOLUTION)이 컨펌 게이트로 수행 — 회고가 직접 공유 룰북을 건드리지 않는다.

## 5. 세션 종료 (컨텍스트 한도)

EX-9 트리거 → **HANDOFF-WRITER 자동 호출** (ERROR-POLICY.md §4 본격 명세).

### 5.1 세션 vs 운영 모드 분리

- 세션 종료: *채팅 세션*만 끝남. 사이클은 진행 중이거나 정지 상태로 유지
- 운영 모드: *시스템 자체*는 살아있음. 다음 세션이 핸드오프로 인계받아 *같은 사이클 재개* 가능

### 5.2 처리 흐름

```
[EX-9 컨텍스트 한도 임박 감지]
       ↓
[HANDOFF-WRITER 호출 — 자동]
   (오케스트레이터의 책임 3 호출 흐름)
       ↓
[HANDOFF-WRITER가 핸드오프 산출]
   - 위치: dlc-meta/cycles/{cycle-id}/handoff-v{n}.md
   - 포맷: HANDOFF.template.md 변수 채움
       ↓
[사용자에게 핸드오프 다운로드 안내]
       ↓
[세션 종료 — 운영 모드는 살아있음]
       ↓
[다음 세션이 핸드오프 첨부 → 같은 사이클 재개]
```

### 5.3 사이클은 종료되지 않음

세션 종료 시 *진행 중 사이클은 PAUSE 상태로 유지*. CYCLE-CLOSER 호출 X. 다음 세션이 사이클 ID 기반 재개.

## 6. 운영 모드 종료

운영 모드는 **기본 영구**. 명시 종료 없으면 *기본 영구 대기*.

### 6.1 종료 신호

| 신호 | 처리 |
|---|---|
| 사용자 명시 종료 ("시스템 끄자", "그만", "프로젝트 종료") | §6.2 정상 운영 모드 종료 |
| 시스템 자체 폐기 (드뭄) | §6.3 시스템 아카이브 |
| 무응답 장기간 | **§6.4 무시 — 정상 운영 상태로 간주** |

### 6.2 정상 운영 모드 종료

```
[사용자 명시 종료 신호]
       ↓
[재확인 1회 — "정말 운영 모드 종료할까요? 진행 중 사이클은 어떻게?"]
       ↓
[진행 중 사이클 처리 결정]
   - 옵션 a: 정상 사이클 종료 후 운영 모드 종료
   - 옵션 b: 사이클 PAUSE 저장 후 즉시 운영 모드 종료
   - 옵션 c: 취소 (운영 모드 종료 안 함)
       ↓
[최종 상태 저장]
   - 모든 사이클 audit.md 보존
   - ORCHESTRATOR.md 누적 패턴 보존
   - 마지막 운영 종료 시점 기록
       ↓
[종료 보고 — §7 형식]
       ↓
[운영 모드 종료]
```

### 6.3 시스템 자체 폐기

시스템 자체를 더 이상 사용 안 함 — *드물지만* 발생 가능.

처리:
- 메타 레포 (`dlc-meta/`) 아카이브 안내
- 영향 레포의 `aidlc-rules/` 보존 안내 (재사용 가능)
- 본 룰북 (`ix-ai-dlc/`) 클론은 그대로 두어도 됨 (다른 시스템에서 재사용)

### 6.4 무응답 장기간 — 정상 운영 상태

**명시 종료 없으면 *기본 영구 대기*.** 응답 없는 기간 = 정상 운영 상태로 간주.

사례:
- 퇴근 후 야간·아침
- 주말
- 휴가·휴지 (며칠~몇 주)
- 장기 휴지 (몇 달)

→ 자동 아카이브 **하지 않음**. 다음 인텐트 수신 시 즉시 운영 모드 진입.

### 6.5 운영 모드 재개

시스템 자체 폐기 후 재개 가능:
- 메타 레포 아카이브 복원
- 마지막 운영 종료 시점부터 이어서 진행 (큐, audit.md, 누적 패턴 모두 복원)

## 7. 종료 시 사용자 보고 형식

종료 단위 따라 보고 형식 다름.

### 7.1 사이클 종료 보고 (정상)

```
✓ 사이클 종료 ({cycle-id})

처리한 인텐트: {원본 인텐트 요약}
영향 레포: {레포 리스트}
주요 산출물:
  - {파일 경로} — {역할}
  - ...
결정 추적: {Decision Log 발췌 (분기점 결정만)}
사용자 컨펌 응답: {주요 응답 요약}

audit.md: dlc-meta/cycles/{cycle-id}/audit.md
다음 권장: {다음 인텐트 후보 또는 대기}
```

### 7.2 사이클 종료 보고 (예외)

```
⚠ 사이클 PAUSE 또는 ABORT ({cycle-id})

종료 사유: {EX-N 또는 사용자 신호}
처리한 인텐트: {원본 인텐트 요약}
완료 진행률: {0~100%}
부분 산출물:
  - {파일 경로} — {상태 (완료/부분)}
  - ...

audit.md: dlc-meta/cycles/{cycle-id}/audit.md
재개 방법: {PAUSE} 사이클 ID 기준 재개 가능 / {ABORT} 새 사이클 권장
```

### 7.3 세션 종료 보고 (EX-9)

```
⏸ 세션 종료 — 컨텍스트 한도 임박

진행 중 사이클: {cycle-id} (PAUSE)
완료 진행률: {0~100%}
핸드오프 산출: dlc-meta/cycles/{cycle-id}/handoff-v{n}.md

다음 세션 시작 가이드:
1. 핸드오프 파일 첨부
2. "사이클 {cycle-id} 재개" 명시
3. 핸드오프의 새 세션 첫 한마디 가이드 따라 진행
```

### 7.4 운영 모드 종료 보고

```
■ 운영 모드 종료

시스템: {시스템 이름}
운영 기간: {시작일} ~ {종료일}
완료 사이클 수: {N}
최종 상태:
  - 활성 레포: {N}개
  - 메타 레포: dlc-meta/
```
