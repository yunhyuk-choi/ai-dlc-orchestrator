## 1. 단계 [1] — 인텐트 유형 분류

7 유형 휴리스틱. *모호 시 사용자 컨펌* (DP-1).

| 유형 | 트리거 예시 | 라우팅 결과 |
|---|---|---|
| **시스템 부트스트랩** | "시스템 새로 시작" / 부트스트랩 미완료 감지 | SETTER 호출 |
| **새 기능 개발** | "새 기능", "X 추가", "구현하자" | SYSTEM-WORKFLOW STEP 0~10 풀 사이클 |
| **변경·수정** | "고쳐", "리팩토링", "버그", "수정" | STEP 1~9 부분 사이클 (STEP 0 스킵, STEP 10 종료 로그는 수행) |
| **분석·조사** | "어떻게 돼있어", "분석", "보고서", "현황" | STEP 1·2만, 코드 변경 없음 |
| **진화·메타** | "룰북 갱신", "패턴 추가", "프로세스 바꿔" / **오케스트레이터 능동 트리거 가능 (DP-9)** | EVOLUTION 분기 (Stage 3 큐) |
| **추가 레포 필요** | "새 레포" / 분석 결과 추가 레포 필요 판단 | REPO-CREATOR 호출 (DP-7) |
| **사이클 종료** | "마무리", "다 됐어", "끝내자" | CYCLE-CLOSER 호출 (DP-8) |

**오케스트레이터 능동 트리거 (진화·메타)**:
- 누적 사이클 로그·반복 패턴 감지 시 사용자에게 *갱신 추천 제안*
- 휴리스틱 본문은 Stage 3 큐 — 본 단계에서는 *분류 가능성*만 명시

**분류 우선순위**:
1. 부트스트랩 상태 체크 (시스템 미부트스트랩 → 무조건 부트스트랩 유형)
2. 명시적 키워드 매칭 ("새 레포" / "끝내자" 등 강한 신호)
3. 자연어 의미 분류 (LLM 판단)
4. 모호 시 사용자 꼬리질문

## 2. 단계 [2] — 영향 범위 분석

REPO-MAP 활용 (원칙 6 단일 원천).

### 2.1 조회 절차

```
[1] 인텐트에서 도메인 키워드 추출
       ↓
[2] REPO-MAP 도메인 컬럼 정방향 조회
   (인텐트 키워드 → 영향 도메인 목록)
       ↓
[3] REPO-MAP 도메인 컬럼 역방향 조회
   (도메인 → 영향 레포 목록)
       ↓
[4] 의존 컬럼 전이적 탐색
   (직접 영향 + 의존 그래프 따라 간접 영향)
       ↓
[5] 영향 레포 리스트 + 도메인 매핑 + 의존 그래프 산출
```

### 2.2 산출물

- **영향 레포 리스트**: 슬러그[]
- **도메인 매핑**: { 도메인 → [레포 슬러그] }
- **의존 그래프**: DAG (노드=레포, 엣지=의존)
- **영향 분류**: 직접 영향 / 의존 영향 / 무영향

### 2.3 DP-2 컨펌 조건

큰 영향 (5+ 레포 또는 의존 root 변경) → 사용자에게 분석 결과 보고 + 컨펌 요청.

## 3. 단계 [3] — SYSTEM-WORKFLOW STEP 선택 (AWS 6 factors)

`ai-dlc-orchestrator/specs/SYSTEM-WORKFLOW.md`의 STEP 0~10 (시스템 레벨 재포지셔닝). 인텐트·영향·복잡도로 *런타임 판단*해 STEP 스킵·추가·detail level 결정.

### 3.1 AWS 6 factors 정의

| Factor | 정의 | 평가 신호 |
|---|---|---|
| **Request Clarity** | 요청 명확성 | 모호한 단어 빈도, 사용자 추가 질문 필요성, 멀티초이스 답변 명확성 |
| **Problem Complexity** | 문제 복잡도 | 도메인 수, 의존 그래프 깊이, 외부 의존성 |
| **Scope** | 작업 범위 | 영향 레포 수, 변경 코드 라인 수 추정, 인터페이스 변경 유무 |
| **Risk Level** | 위험도 | 비가역성, 운영 중 시스템 영향, 보안·컴플라이언스 |
| **Available Context** | 가용 컨텍스트 | greenfield/brownfield, 기존 산출물 유무, REPO-MAP 매핑 완성도 |
| **User Preferences** | 사용자 선호 | 명시 의사 (빠른 진행 / 신중) / 누적 선호 패턴 (ORCHESTRATOR.md EVOLUTION) |

### 3.2 STEP 적용 가이드 (인텐트 유형별 디폴트)

| 인텐트 유형 | 디폴트 STEP | 6 factors 영향 |
|---|---|---|
| 시스템 부트스트랩 | SETTER 호출만 (STEP 무관) | — |
| 새 기능 개발 | STEP 0~10 풀 | Complexity ↑ → detail level ↑ |
| 변경·수정 | STEP 1~9 (STEP 0 스킵, STEP 10 종료 로그 수행) | Risk ↑ → STEP 6(통합 테스트) 강제 |
| 분석·조사 | STEP 1·2만 | Scope 따라 STEP 1 깊이 조정 |
| 진화·메타 | EVOLUTION 분기 (별도) | — |
| 추가 레포 필요 | REPO-CREATOR 호출 후 풀 사이클 | — |
| 사이클 종료 | CYCLE-CLOSER 호출 (STEP 무관) | — |

### 3.3 detail level 결정

각 STEP의 *깊이*를 6 factors로 조정:
- Clarity 낮음 → 사용자 인터뷰 단계 늘림
- Complexity 높음 → 설계 단계(STEP 2·3) 깊이 ↑
- Risk 높음 → 통합 테스트(STEP 6) + 빌드 검증(STEP 7) 강화
- Scope 좁음 → STEP 8(보조 자료) 스킵 가능
- Available Context 충분 → STEP 1(분석) 단축

### 3.4 산출물

DP-3 결정 → `aidlc-docs/execution-plan.md` (AWS 호환 형식, 영향 받는 각 레포에 생성):
- 실행 STEP 리스트
- 각 STEP 스킵·실행 이유 (6 factors 평가 결과)
- 각 STEP detail level

사용자 검토·승인 → 단계 [4] 진입.

## 4. 단계 [4] — 호출 대상 결정

작업 유형 → 호출 대상 매핑 (DP-4, 자율).

| 작업 유형 | 호출 대상 |
|---|---|
| 시스템 부트스트랩 | SETTER |
| 추가 레포 생성 | REPO-CREATOR |
| 레포별 셋업 (AWS 룰셋 설치 포함) | REPO-SETTER |
| **CI/CD 부착·통합** (셋업 시 REPO-SETTER 경유 또는 운영 중 온디맨드 재부착) | **CICD-SETTER** (조건부 — POLICY-RELEASE) |
| **환경 배포·착지 후 배포** (검증된 릴리스 이후) | **DEPLOY-AGENT** (조건부 — POLICY-DEPLOY; non-prod 자율 / prod 사람게이트 A-4) |
| **cross-repo 설계·통합·검토** | **오케스트레이터 직접** (서브 없이 수행) |
| **단일 레포 내부 워크플로우** (요구사항·설계·코드·테스트·빌드) | **그 레포 AWS AI-DLC 에이전트** |
| 핸드오프 문서 생성 | HANDOFF-WRITER |
| 사이클 로그 기록 | CYCLE-LOG (룰북 따라 직접 기록) |
| 사이클 종료 후처리 | CYCLE-CLOSER |
| 사용자 결정 필요 시점 | 사용자 (멀티 초이스) |

### 4.1 오케스트레이터 직접 수행 vs 위임 기준

**직접 수행**:
- cross-repo 결합점 검증 (단일 레포 책임 범위 초과)
- 시스템 레벨 의사 결정 (라우팅, 분기, 통합)
- 사용자 보고·인터페이스

**위임**:
- 단일 레포 내부 작업 (AWS 위탁)
- 표준화된 부트스트랩·셋업 작업 (우리 서브)
- 산출물 생성 (HANDOFF-WRITER 등)
- **CI/CD 부착·통합 (조건부)** — CICD-SETTER. 레포가 CI/CD를 가져야 할 때만(REPO-SETTER 셋업 시 또는 운영 중 재부착 신호). 부착할 표면이 없으면 호출 안 함.
- **환경 배포 (조건부)** — DEPLOY-AGENT. *검증된 릴리스 이후* 착지 후 배포 단계(SYSTEM-WORKFLOW STEP 9.5). non-prod는 자율, **production은 사람 게이트(A-4)**. 배포 표면이 없으면 호출 안 함.

## 5. 단계 [5] — 호출 순서·병렬성

오케스트레이터가 *능동 영향 분석*으로 결정 (DP-5).

### 5.1 알고리즘

```
[1] 단계 [2] 산출물 가져오기
   (영향 레포 리스트 + 의존 그래프)
       ↓
[2] 의존 그래프 위상 정렬
   (의존 root 가까운 레포부터 순서)
       ↓
[3] 무영향 그룹 식별 (클러스터링)
   기준: 의존 없거나 이번 작업에서 서로 영향 없음
       ↓
[4] 같은 클러스터 = 병렬 호출 가능
   다른 클러스터 = 위상 순서 따라 직렬
       ↓
[5] 사용자 컨펌 지점은 *직렬*
   (병렬 컨펌은 인지 과부하)
       ↓
[6] DAG 실행 계획 산출
   → AWS execution-plan.md 호환 형식
```

### 5.2 클러스터링 기준

같은 클러스터 (병렬 가능):
- REPO-MAP 의존 컬럼에 서로 없음
- 이번 작업 영향이 *동일 인터페이스 미공유*
- cross-repo 결합점 미생성

다른 클러스터 (직렬):
- 의존 관계 있음 (의존 root 먼저)
- 동일 인터페이스 변경 (의존받는 쪽 먼저)
- 결합점 신규 생성 (한쪽 먼저 정의)

### 5.3 산출물

DP-5 결정 → DAG 실행 계획. 형식:
- 노드: (레포 슬러그, STEP, 호출 대상, detail level)
- 엣지: 의존 관계 (실행 순서 제약)
- 클러스터: 동시 실행 그룹 ID

AWS `execution-plan.md` 호환 형식으로 직렬화.

### 5.4 사용자 컨펌 (execution-plan.md 게이트)

DP-3 + DP-5 결과를 통합 `execution-plan.md`로 산출 → 사용자 검토 → 승인 시 단계 [6] 진입.

거부 시 → 단계 [3] 또는 [5]로 회귀 + 재평가.

## 6. 단계 [6] — 실행 + 결과 통합

### 6.1 실행 절차

```
[1] DAG 따라 호출 (병렬 그룹 동시, 직렬 단계 순차)
       ↓
[2] 각 호출 응답 대기
   - 우리 서브: 4-튜플
   - AWS 에이전트: aidlc-state.md 진행 상태 모니터링
       ↓
[3] 응답 검토 (DP-6)
   - 정합성 체크 (서브 4-튜플 두 번째 필드)
   - 성공·실패 여부 (AWS audit.md)
   - cross-repo 결합점 검증 (오케스트레이터 직접)
       ↓
[4] 다음 분기 판단
   - 정상 → 다음 STEP 또는 다음 사이클 반복
   - 예외 → ERROR-POLICY.md 정책 적용
   - 큰 변화 → 사용자 컨펌 (DP-6)
```

### 6.2 cross-repo 결합점 검증

- 인터페이스 일치 확인 (API 시그니처, 데이터 형식)
- 의존 방향 일관성
- 통합 테스트 실행 (STEP 6 — 시스템 레벨)
- 실패 시 → EX-8 (cross-repo 결합점 검증 실패) → 영향 레포 재호출

### 6.3 결과 통합 (audit.md 흡수)

각 레포의 `aidlc-docs/audit.md` → 시스템 단위 `dlc-meta/cycles/{cycle-id}/audit.md`로 통합 집계 (책임 7). 형식·필드는 CYCLE-LOG.md 본문 위임.

## 부록. 6 factors 평가 휴리스틱 (예시)

LLM 판단을 보조하는 평가 신호 가이드. *엄격한 점수화 아님* — 정성 평가.

| Factor | 낮음 | 중간 | 높음 |
|---|---|---|---|
| Clarity | 사용자 모호한 자연어, 추가 질문 3+ 필요 | 일부 모호, 1~2 질문 | 명확한 인텐트, 추가 질문 불필요 |
| Complexity | 단일 도메인, 의존 없음 | 2~3 도메인, 의존 1~2 계층 | 다중 도메인, 의존 root 영향 |
| Scope | 단일 레포, 변경 작음 | 2~4 레포 | 5+ 레포 또는 시스템 전체 |
| Risk | 가역적, 격리된 변경 | 일부 비가역 | 비가역·운영 영향·보안 |
| Available Context | greenfield, 산출물 없음 | brownfield 일부 매핑 | brownfield 완전 매핑, 기존 산출물 풍부 |
| User Preferences | 미명시 | 일부 명시 ("빠르게" 등) | 명시적 + 누적 패턴 일치 |
