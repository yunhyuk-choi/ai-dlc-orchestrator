## 1. 분기점 카탈로그 (DP-1 ~ DP-10)

각 분기점의 위치·입력·결정 기준·컨펌 정책·출력·기록 형식.

## 2. 사용자 컨펌 vs 자율 판단 기준

### 2.1 자율 판단 영역 (오케스트레이터 단독)

- **가역적·내부적 결정** — 분류 (DP-1 명확), 매핑 (DP-2 단순), 정렬 (DP-5 의존 명확), 응답 검토 (DP-6 정상)
- **작은 영향 범위** — 단일 레포, 소규모 변경
- **명확한 입력** — 추가 질문 불필요

### 2.2 사용자 컨펌 필수 영역 (4 카테고리)

1. **비가역적 결정**
   - 신규 레포 생성 (DP-7)
   - 룰북 갱신 (DP-9)
   - 사이클 종료 (DP-8 중 일부)
   - 외부 커밋·머지·릴리스
   
2. **큰 영향 범위**
   - 5+ 레포 영향 (DP-2)
   - 의존 root 변경 (DP-2)
   - 인터페이스 변경 (DP-6)

3. **모호한 입력**
   - 인텐트 분류 신뢰도 낮음 (DP-1)
   - 자연어 파싱 실패 (EX-4)
   - 휴리스틱 mismatch (EX-10)
   - → 꼬리질문으로 해소 (ORCHESTRATOR-AGENT.md §3.4 표준)

4. **AWS execution-plan.md 검토 게이트**
   - DP-3 + DP-5 결과 통합 검토
   - AWS 에이전트 트리거 직전 필수 게이트

> 위 4 카테고리가 *구체 결정 상황*에 어떻게 매핑되는지는 **§5 결정 아키타입**이 보여준다 (예: 외부·비가역 행위 → A-4 = 카테고리 1, 공유 계약 변경 → A-1 = 카테고리 2, 기준값 정규화 → A-2 = 카테고리 3).

### 2.3 컨펌 방식 표준

| 컨펌 유형 | 형식 |
|---|---|
| 단순 OK/NO | 멀티초이스 (옵션 2) |
| 복수 선택 | 멀티초이스 (옵션 3+) |
| 자유 응답 | `[Answer]:` 태그 |
| 꼬리질문 묶음 | 한 메시지에 직접 연결된 후속 질문 함께 (인지 과부하 방지 원칙) |

## 3. 결정 기록 (audit.md 호환)

**모든 분기점 결정은 audit.md에 기록.** DP-9 진화 제안의 데이터 소스가 풍부해짐.

### 3.1 기록 위치

```
dlc-meta/cycles/{cycle-id}/audit.md
```

append-only. 사이클 단위 디렉토리. 형식 명세는 `CYCLE-LOG.md` 위임.

### 3.2 분기점 기록 표준 형식

```
[YYYY-MM-DD HH:MM:SS] DP-N: {분기점 이름}
  Input: {입력 요약}
  Decision: {결정 + 이유}
  Confirm: {사용자 컨펌 여부 + 응답 ([Answer]: 태그 원문)}
  Output: {출력 요약}

```

### 3.3 예외 기록 형식

ERROR-POLICY.md §3 참조. 분기점 기록과 동일 형식 + 예외 필드 (EX-N + 적용 정책 + escalate 여부).

### 3.4 진화 제안 (DP-9) 데이터 소스 흡수

DP-9 입력 = audit.md 누적 entry. 패턴 감지 휴리스틱이 다음을 본다:
- 동일 분기점 반복 빈도
- 동일 사용자 컨펌 응답 패턴
- 동일 예외 반복
- 동일 호출 경로 반복

→ Stage 3에서 본격 휴리스틱 정의.

## 5. 결정 아키타입 (Decision Archetypes)

§1 DP-1~10은 *어디서·무엇을* 결정하는지의 **추상 카탈로그**다. 그러나 실제 사이클에서 반복적으로 마주치는 결정들은 더 **구체적인 형태**를 띤다 — "이 수정을 소비처에 둘까 공유 라이브러리에 둘까?", "둘 중 무엇을 단일 원천으로?", "기본 브랜치에 직접 넣을까 MR/PR로?" 등. 본 절은 그런 **재발 결정의 구체 아키타입**을 정의한다.

각 아키타입은 하나 이상의 DP-N을 **특수화**(specialize)한다 — DP가 *결정의 자리*라면, 아키타입은 그 자리에서 *반복되는 결정의 모양*이다. 아키타입은 **프로젝트 무관**이며, 어떤 다중 레포 소프트웨어 프로젝트에도 일반화된다 (괄호 안 예시는 *예시일 뿐 규칙이 아님*).

각 아키타입 명세 항목: **언제 발생** / **선택지** / **기본·권장** / **컨펌 vs 자율 기준** / **특수화하는 DP**.

> 컨펌 기준은 §2.2(컨펌 필수 4 카테고리)와 정합한다. 아키타입은 그 카테고리를 *구체 상황*에 매핑할 뿐, 새 컨펌 정책을 만들지 않는다.

### A-1 — 수정 위치: 소비처/앱 vs 공유 라이브러리/생산자

| 항목 | 내용 |
|---|---|
| 언제 발생 | 어떤 결함·변경이 **그것을 소비하는 앱**에도, **그 앱이 의존하는 공유 상류(upstream)**에도 존재할 수 있을 때 (예: 공유 UI 컴포넌트의 결함이 앱에서 드러남) |
| 선택지 | ① 소비처/앱에 로컬 수정 ② 공유 라이브러리/생산자에 상류 수정 (모든 소비처가 혜택) ③ 상류 수정 + 앱은 임시 로컬 우회 후 채택 시 제거 |
| 기본·권장 | **판별 기준으로 결정**: 공유 컴포넌트의 *진짜 결함 또는 공통 표준*이면 → **상류 수정**(모든 소비처 이익). 앱 고유 스타일·동작이면 → **로컬 유지**. 불명확하면 상류로 기울이되 컨펌 |
| 컨펌 vs 자율 | **공유 계약(contract)을 바꾸면 컨펌** (시그니처·동작·토큰 등 — §2.2-2 인터페이스 변경). 순수 로컬·가역 수정은 자율 |
| 특수화 | **DP-2**(영향 범위 — 상류 수정은 역방향 소비처로 전이) + **DP-6**(cross-repo 결합점 검증) |

### A-2 — 기준값 정규화: 둘 이상의 원천이 어긋날 때 무엇을 단일 원천으로

| 항목 | 내용 |
|---|---|
| 언제 발생 | 같은 사실을 표현하는 **둘 이상의 원천이 불일치**할 때 (예: 코드 vs 디자인, 또는 두 설정 파일, 또는 스키마 vs 문서) |
| 선택지 | ① 원천 X를 정규(canonical)로, Y를 X에 동기화 ② 그 반대 ③ 제3의 합의값을 새 정규로 ④ 의도된 분기로 판정해 동기화 안 함 |
| 기본·권장 | **기본 디폴트 없음** — 무엇이 정규인지, **동기화 방향**, **그 원천의 소유자(owner)**를 명시하는 것이 본질. 원칙 6(단일 원천)에 비추어 *권위 있는 원천*을 정규로 |
| 컨펌 vs 자율 | **항상 컨펌** — 어느 쪽이 옳은지는 사람·소유자의 판단 영역 (§2.2-3 모호 입력에 준함). 자동 판정 금지 |
| 특수화 | **DP-6**(정합성 체크에서 불일치 감지) + **DP-2**(정규화가 양쪽 원천의 레포에 영향) |

### A-3 — 릴리스 방식: 기본 브랜치 직접 vs MR/PR

| 항목 | 내용 |
|---|---|
| 언제 발생 | 변경·버전 범프를 **착지(land)**시킬 때 — 어떤 경로로 기본 브랜치에 반영할지 |
| 선택지 | ① 기본 브랜치 직접 push ② MR/PR 생성 후 리뷰·머지 |
| 기본·권장 | **레포의 브랜치 보호 규칙 + 팀이 명시한 컨벤션**으로 결정. 행위자가 기본 브랜치를 직접 push할 수 있는가? → 문서화된 릴리스 프로세스를 **기본**으로 따름. 보호 규칙·관례가 리뷰를 요구하면 → **MR/PR** |
| 컨펌 vs 자율 | 경로 자체는 정책 기반이라 자율 가능하나, **머지·릴리스라는 행위는 외부·비가역 → A-4 적용**(컨펌). 보호 규칙 불명확 시 컨펌 |
| 특수화 | **DP-3/DP-5**(execution-plan 게이트에서 착지 방식 포함) + **DP-8**(사이클 종료 직전 착지) |

### A-4 — 외부·비가역 행위 승인

| 항목 | 내용 |
|---|---|
| 언제 발생 | **외부로 나가거나 되돌릴 수 없는 행위** 직전 (예: publish/deploy/외부 전송/삭제/force-push/머지) |
| 선택지 | ① 행위 실행 ② 보류하고 컨펌 요청 ③ 드라이런·시뮬레이션만 |
| 기본·권장 | **항상 사전 컨펌** — 일반적 "진행해"가 깔려 있어도. 그 *특정 행위*에 대해 **지속적·명시적으로** 위임된 경우에만 자율 |
| 컨펌 vs 자율 | **일반 "계속"은 비가역 외부 행위의 포괄 위임이 아님**. 특정 행위·범위에 대한 명시 승인만 자율을 허용 (§2.2-1 비가역적 결정의 구체화) |
| 특수화 | **DP-7**(비가역 = 항상 컨펌과 동일 원리) + **DP-8**(종료 시 외부 착지) + **DP-10**(권한·정책 예외와 연계) |

### A-5 — 생산자→소비자 채택 순서

| 항목 | 내용 |
|---|---|
| 언제 발생 | 공유 컴포넌트 변경을 **소비자가 채택하려면 먼저 릴리스되어 있어야** 할 때 (생산자 미릴리스 상태에서 소비자부터 바꾸면 *동작 중인 소비자가 깨짐*) |
| 선택지 | ① 생산자 릴리스 → 소비자 채택 (정순) ② 소비자 먼저 (금지 — 깨짐) ③ 임시 핀·우회로 디커플 후 릴리스되면 정렬 |
| 기본·권장 | **정순 고정**: 생산자 변경을 *릴리스 게이트* 통과시킨 뒤 소비자가 그 릴리스 버전을 채택. DP-5 위상 정렬에서 생산자 노드가 소비자 노드보다 선행 |
| 컨펌 vs 자율 | 순서 자체는 **자율**(의존 그래프로 결정 — §2.1). 단, 생산자 *릴리스* 단계는 A-3·A-4가 컨펌을 요구할 수 있음 |
| 특수화 | **DP-5**(호출 순서·병렬성 — 릴리스 게이트 선행 제약) + **DP-2**(생산자→소비자 전이). *Cross-ref*: `SYSTEM-WORKFLOW.md` STEP 9(다중 레포 커밋·머지)의 릴리스 게이트 |
