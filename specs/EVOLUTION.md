## 3. 신호별 휴리스틱 본문

### 3.A 신호 ② — 컨펌 응답 패턴 (파일럿)

#### 감지 (STEP 10 직후)

입력: `index.md` B-② — DP별 최근 K=5 사이클 컨펌 응답 시퀀스.

DP별 산출:
- `n` = 시퀀스 길이 (해당 DP가 윈도우 내 컨펌된 횟수, ≤ K)
- `a*` = 최빈 응답, `m` = `a*`의 횟수

#### 후보화 조건

```
후보 = (m ≥ 3)            ← 빈도 floor (충분한 증거, 우연 배제)
       AND (m / n ≥ 0.8)  ← 일관성 (해당 DP 컨펌 중 80%+ 동일)
```

- `n < 3`이면 floor 미달 → 후보 아님
- 예: 시퀀스 `승인,승인,승인,거부,승인` → n=5, a*=승인, m=4 → 4≥3 ✓, 4/5=0.8 ✓ → **후보**
- 예: `승인,승인,거부` → n=3, m=2 → 2<3 → 후보 아님

#### 제안 내용

후보 DP에 대해 §2 매핑 적용 — 둘 중 택1을 사용자에게 제시. **적용 대상 = 인스턴스 `dlc-meta/ORCHESTRATOR.md`의 컨펌 오버라이드 섹션** (공유 `DECISIONS.md` 컨펌 디폴트는 불변 — 프로젝트별 학습이 전 프로젝트에 새지 않게, D-Stage3-6):
- (a) 그 DP의 컨펌을 **자율 전환** (원칙 4 — 이후 `a*`로 자동 처리, 묻지 않음)
- (b) 컨펌은 유지하되 **디폴트 선택을 `a*`로** (사용자는 바꿀 때만 개입)

### 3.B 신호 ① — 레포 공동 영향

#### 감지 (STEP 10 직후 — 로그만)

입력: `index.md` B-① — 레포 쌍별 최근 K=5 사이클 공동 출현 집계 (DP-2 영향 매핑 Output에서 추출).

쌍별 산출:
- `M` = 공동 출현 사이클 수
- `N` = 윈도우 내 DP-2 발생 사이클 수

*감지는 로그(B-①)만 본다. REPO-MAP은 감지 입력 아님.*

#### 후보화 조건

```
후보 = (M ≥ 3) AND (M / N ≥ 0.8)
       AND 그 레포 쌍이 REPO-MAP에 이미 의존으로 선언돼 있지 않음
```

- `M`·일관성 임계는 ② 재사용 (3회/80%).
- **마지막 조건은 감지가 아니라 *제안 형성 단계의 중복 필터*** — 이미 선언된 의존이면 새 정보가 아니므로 제안 안 함. REPO-MAP은 여기서 *현재 선언 상태 대조용*으로만 읽힘 (패턴은 어디까지나 로그에서 나옴).

#### 제안 내용 + 영향

- 제안: REPO-MAP에 두 레포 간 의존 명시 추가 (방향은 추정치 제시 + 사용자 확인).
- **영향 (중요)**: REPO-MAP 변경은 *향후 모든 DP-2 영향 매핑*을 바꾸는 **고레버리지 변경** — ②(단일 DP 컨펌 정책)보다 파급이 큼. 영향 분석에 "향후 영향 범위 변화"를 반드시 명시. ①은 ②보다 보수적 임계를 둘 수도 있음 (큐 — 파일럿 후 조정).

### 3.C 신호 ③ — 예외 반복

#### 감지 (STEP 10 직후 — 로그만)

입력: `index.md` B-③ — (EX-N, 레포)별 최근 K=5 사이클 발생 집계 (audit.md 예외 기록 §7에서 추출, repo-scoped 예외만).

(EX-N, 레포)별 산출:
- `M` = 해당 예외 발생 사이클 수
- `N` = 윈도우 내 그 레포가 관여한 사이클 수 (DP-2 영향 리스트 기준)

#### 후보화 조건

```
후보 = (M ≥ 3) AND (M / N ≥ 0.8)   ← 그 레포 관여 시 80%+ 재발 = 체계적
```

- ②/① 임계 재사용. 단 여기서 `M/N`은 **재발률** — 같은 예외가 그 레포를 건드릴 때마다 거의 매번 터지는가.
- 예외는 *대안 결과가 없으므로* ②식 "동일 응답 일관성"이 아니라 *재발률*로 해석.

#### 제안 내용 + 영향

- §2 매핑 — 예외 성격 따라 택1. **적용 대상 = 인스턴스 `dlc-meta/ORCHESTRATOR.md`의 예외 오버라이드 섹션** (공유 `ERROR-POLICY.md`는 디폴트로 불변, D-Stage3-6):
  - (a) **예외 특화 (인스턴스)** — 그 (EX, 레포)에 특화 처리 룰 추가 (재시도 전략 변경·escalate 경로 조정 등). 공유 ERROR-POLICY 디폴트 위에 *프로젝트 오버라이드*로 얹음
  - (b) **REPO-SETTER 재점검** — 설치/버전 문제 의심 시(예: EX-3 버전 mismatch 반복) 그 레포 셋업 재검증 권고 (일회성 액션 — 룰 변경 아님)
- **영향**: ①보다 *국소적* — ERROR-POLICY 특화는 해당 예외에 한정, REPO-SETTER 재점검은 일회성 교정. 의존 그래프(①)·라우팅(④)처럼 시스템 전반에 파급되지 않음.

### 3.D 신호 ④ — 호출 경로 반복

#### 감지 (STEP 10 직후 — 로그만)

입력: `index.md` B-④ — 인텐트 시그니처별 정규 호출/STEP 시퀀스 집계 (audit.md 라우팅 트레이스에서 추출).

시그니처별 산출:
- `M` = 동일 시퀀스 반복 사이클 수
- `N` = 그 인텐트 시그니처 사이클 수
- **시그니처 입도 디폴트 (D-Stage3-5)** = `(인텐트 유형, 영향 도메인 집합)`:
  - **인텐트 유형** = DP-1 분류값 (ROUTING §1 — 7유형).
  - **영향 도메인 집합** = ROUTING §2.2 산출물 *도메인 매핑*의 도메인 키 집합 (중복제거·정렬). REPO-MAP에서 *역파생하는 게 아니라* 영향 분석(단계 2)이 그 사이클에 산출한 *일차 값* — audit.md DP-2 기록에 영속화되므로 `index.md` 재생성 시에도 불변 (**네이티브 스냅샷** — 도메인 재라벨에 의한 재버킷 없음).
  - **입도 근거**: 정확한 *레포 집합* 입도는 매 사이클 미세 변동으로 `M≥3` floor를 거의 못 넘김(과적합 → 신호 사문화). *도메인 집합*은 "같은 종류의 작업"이 같은 도메인 묶음으로 반복돼 floor 도달 + 경로 일관성 의미 보존. (①과의 직교는 *출력 측*에서 이미 보장 — ①=REPO-MAP 의존, ④=ORCHESTRATOR.md 라우팅 — 이므로 입도로 직교를 만들 필요는 없음.)
  - **키/값 분리**: 시그니처 *키*(유형·도메인)는 DP-1·DP-2 §6 기록만으로 *지금* 산출 가능. *값*(정규 호출 시퀀스)만 audit.md 라우팅 트레이스(§8 PROGRESS 정식화) 전제에 잔존.

#### 후보화 조건

```
후보 = (M ≥ 3) AND (M / N ≥ 0.8)   ← 같은 인텐트 → 같은 경로 80%+ = 안정 라우팅
```

#### 제안 내용 + 적용 경계 + 영향

- 제안: 그 (시그니처 → 시퀀스)를 **라우팅 단축(매크로)**로 등록.
- **적용 위치 = `ORCHESTRATOR.md` "누적 라우팅 패턴" 섹션** — 승인된 진화가 반영되는 *결과 측*. 형식은 `ORCHESTRATOR.template.md` 해당 섹션.
- **입출력 경계 재확인**: **감지는 로그(B-④), 적용은 ORCHESTRATOR.md.** ②①③은 적용도 로그/룰 파일이었지만, ④만 적용 결과가 오케스트레이터 라우팅이라 ORCHESTRATOR.md를 건드림 — 단 *입력*으로 읽는 게 아니라 *승인된 결과를 쓰는* 것 (관심사 분리, D-Stage3-1).
- **상태 = advisory**: 등록된 단축은 *디폴트 힌트*일 뿐 런타임이 여전히 재평가 가능 — 라우팅 경직·과적합 방지.
- **영향**: 라우팅 행동을 바꾸는 고레버리지(①급)지만 advisory라 가역적. 영향 분석에 "어느 인텐트군 라우팅이 바뀌는지" 명시.
- **전제**: 호출 경로가 audit.md 라우팅 트레이스로 기록돼야 함 (CYCLE-LOG B-④ 전제).

### 3.E 신호 ⑤ — 회고 신호 (서술적, TERMINATION §4.4 산출)

②①③④가 *로그를 세어* 패턴을 잡는다면, ⑤는 *회고가 글로 지목한* 룰북 갭·반복 마찰·검증 미스를 진화 입력으로 받는다. 빈도 임계로 안 잡히는 *단발이지만 중요한* 교훈(예: "이 룰북에 X 절차가 없어 헤맴", "환경 gotcha를 또 밟음")을 1급 신호로 승격.

#### 입력 (STEP 10 직후 — 이번 사이클 회고)

입력: `cycles/{cycle-id}/retrospective.md` (TERMINATION §4.4 산출, `templates/RETROSPECTIVE.template.md` 채움). 핵심 슬롯:
- **룰북/프레임워크 갭** + *각 갭을 메울 파일* (회고가 직접 지목)
- **제안 룰북 변경** (→ 본 EVOLUTION 입력으로 명시 표시된 슬롯)
- **검증 미스** (정합성 체크가 놓친 것)
- **(재)발견된 환경 gotcha**

*감지 입력은 회고 텍스트 그 자체*. 로그 롤업(index.md)을 보지 않는다 — ②①③④와 직교한 보강 채널.

#### 후보화 조건

②①③④의 *빈도/일관성 임계(3회·80%)를 적용하지 않는다*. 회고 신호는 카운팅이 아니라 *지목*이므로:

```
후보 = 회고의 "제안 룰북 변경" 슬롯이 비어있지 않음
       AND 지목된 변경이 구체적 파일·섹션을 가리킴 (모호한 불만 X)
```

- 단발 회고도 후보가 될 수 있음 (질적 신호). 단 *반복되면 가중* — 같은 갭이 K=5 윈도우 내 복수 회고에 등장하면 우선순위 상향(②식 빈도가 *보조 가중치*로만 작동, floor 아님).

#### 적용처 분기 (중요 — 인스턴스 vs 공유)

회고가 *지목한 파일*에 따라 라우팅이 갈린다:
- **인스턴스 귀속** (지목 파일 = `ORCHESTRATOR.md` 컨펌/예외 오버라이드·라우팅 패턴, `REPO-MAP`): ②①③④와 동일 경로 — `dlc-meta` 인스턴스에 §4 게이트로 적용.
- **공유 룰북 귀속** (지목 파일 = 공유 `ai-dlc-orchestrator`의 DECISIONS·ERROR-POLICY·SYSTEM-WORKFLOW·템플릿 등): **자동 적용 금지** (D-Stage3-6 불가침). 대신 *사람-큐레이션 큐*에 제안으로 등재 — 원칙 8 PR/머지 통로(SETTER §6)로만 반영. DP-9 자동 경로 밖.

> 이 분기가 ⑤의 핵심: 회고는 *공유 룰북 갭*을 자주 가리키는데(런타임 로그는 거의 못 가리킴), 그것을 *자동 머지하지 않고* 사람 큐레이션으로 안전하게 흘린다.

#### 영향

- 인스턴스 귀속분: ②③④와 동급 가역성 (dlc-meta 커밋, 컨펌 게이트).
- 공유 귀속분: 시스템 *전 프로젝트*에 파급될 수 있는 최고 레버리지 — 그래서 *자동 아님*, 사람 검토 필수.

#### 전제

회고가 산출돼야 함 (TERMINATION §4.4 — 종료 절차 정식 단계). 사소 사이클은 경량 회고라 ⑤ 후보가 비는 게 정상 (false negative 아님 — 지목할 갭이 없었다는 뜻).

## 4. 제안 출력·게이트

### 4.1 전달 — 보수 (D-Stage3-1)

- 후보가 잡혀도 **사이클 중 끼어들지 않음**. STEP 10(사이클 종료)에 **모아서** 제안 (인지 과부하 방지 원칙).
- 복수 후보 시 한 메시지에 묶어 제시.

### 4.2 출력 포맷 (DECISIONS DP-9 출력 정합)

각 후보당: **구체 패턴**(어느 DP·응답 시퀀스·n/m) + **제안 변경**(a/b) + **영향 분석**(가역성·영향 범위).

### 4.3 게이트 — 항상 컨펌 + 인스턴스 커밋

- DP-9 = **항상 사용자 컨펌** (오케스트레이터 *행동*을 바꾸므로 — 컨펌은 비가역성이 아니라 *사용자 opt-in* 논리. 커밋은 어차피 가역).
- 승인 시 적용 = **`dlc-meta` 인스턴스 갱신** (D-Stage3-6): ②③④ → `ORCHESTRATOR.md`(컨펌/예외 오버라이드·라우팅 패턴) / ① → `REPO-MAP.md`. **공유 `ai-dlc-orchestrator`(DECISIONS·ERROR-POLICY·템플릿)는 건드리지 않음.**
- 적용은 **별도 PR 흐름이 아니라 `dlc-meta`의 루틴 커밋** — 진화 변경은 프로젝트 자기 메타 레포에 동승해 머지됨. *전용 진화-PR 파이프라인 없음.*
- **원칙 8 "PR/머지"는 공유 룰북·템플릿 변경 전용** (예: 어떤 패턴이 *보편적*이라 판단해 공유 `ai-dlc-orchestrator`에 반영) — 이는 *사람의 수동 큐레이션*이며 DP-9 자동 경로 **밖**. SETTER §6 "템플릿 변경은 PR/머지로만"이 그 통로.

## 6. 잔여·튜닝 (4신호 본문 완료 후)

- **4신호(②①③④) 본문 정의 완료 + 회고 신호(⑤) 추가.** ②①③④ = 로그 카운팅, ⑤ = 서술적(회고). 추가 신호는 필요 시 §2에 매핑부터.
- **튜닝 큐**: 신호별 임계 차등화(① 고레버리지 → 더 보수적?), 임계값 운영 데이터 기반 재조정. (④ 시그니처 입도 → **D-Stage3-5로 디폴트 박음**: `(인텐트 유형, 영향 도메인 집합)`. §3.D.)
- **데이터 전제**: ③ = audit.md `Repo` 필드(추가 완료), ④ = 라우팅 트레이스 PROGRESS 기록(권장/전제 — 미기록 시 ④ 미작동).
- **적용 경계**: 모든 신호 감지=로그(index.md). 적용 결과는 전부 **`dlc-meta` 인스턴스** — ②=ORCHESTRATOR.md 컨펌 오버라이드 / ①=REPO-MAP.md / ③=ORCHESTRATOR.md 예외 오버라이드·REPO-SETTER 재점검 / ④=ORCHESTRATOR.md 라우팅 패턴. **공유 룰북 불가침** — 전부 dlc-meta 루틴 커밋 경유, 전용 PR 파이프라인 없음 (D-Stage3-6). 원칙 8 PR/머지는 공유 룰북 사람-큐레이션 전용.
