# CICD-RELEASE-ADAPTER.md — CI/CD·릴리스 머신 통합 인터페이스 명세

> **이 문서는 형식·룰 명세 문서.** 에이전트 룰북 아님.
> 본 명세는 오케스트레이터가 관리 대상 레포의 *CI/CD 파이프라인 + 릴리스·퍼블리시 머신* 과 상호작용하는 *조건부 통합 표준* 을 정의한다.
> 실행 주체는 오케스트레이터(게이트 인식·릴리스 조율·검증 트리거) / 각 서브 에이전트(자가 검증). 본 문서는 *룰만* 정의.
> 본 명세는 **조건부** 다 — 레포가 CI/CD 또는 퍼블리시·릴리스 플로우를 *실제로 가질 때만* 발동하고, 없으면 비활성(inert)이다(§1).
> 도구·기술 명칭(CI 설정 파일·레지스트리·버전 도구·시각회귀 리뷰 도구 등)은 모두 *예시(e.g.)* 일 뿐 요구사항이 아니다. 프로젝트·OS에 맞는 동치 수단으로 대체한다.
> 변경은 깃 PR/머지 (원칙 8) — `specs/SYSTEM-WORKFLOW.md` STEP 9(릴리스 게이트) / `agents/orchestrator/DECISIONS.md` §5(A-3·A-4·A-5) / `agents/orchestrator/ERROR-POLICY.md` EX-13 / `specs/VERIFICATION.md`(POLICY-VERIFY)가 본 명세에 연계되므로 변경 시 영향 검토 필수.
> 위치: `ai-dlc-orchestrator/specs/CICD-RELEASE-ADAPTER.md`

> **POLICY-RELEASE (CI/CD·릴리스 통합)** — 레포가 CI/CD·퍼블리시 플로우를 가질 때, 오케스트레이터는 그 *게이트 집합* 을 착지 전에 이해하고, *필수 게이트 실패는 우회 금지 신호* 로 다루며(사람 승인 게이트는 사람 수락을 대기), 릴리스는 *버전 범프 → 기본 브랜치 착지(A-3) → 태그/릴리스 → CI 퍼블리시 → 레지스트리·엔드포인트에서 산출물 존재 검증(POLICY-VERIFY)* 의 순서를 따른다. 공유 컴포넌트는 *퍼블리시·검증 완료 후에만 소비자가 채택(A-5)* 한다. CI/릴리스 자격증명의 스코프·역할 부족은 EX-13로 진단한다.

## 1. 정체성 / 적용 조건

| 항목 | 내용 |
|---|---|
| 종류 | 형식·인터페이스 명세 문서 (에이전트 X) |
| 실행 주체 | 오케스트레이터 (게이트 인식·릴리스 조율·검증 트리거) / 각 서브 에이전트(자가 검증). 본 룰의 *CI/CD 부착·통합* 실행은 `agents/CICD-SETTER.md` |
| 참조 주체 | `specs/SYSTEM-WORKFLOW.md` STEP 9 / `agents/orchestrator/DECISIONS.md` §5(A-3·A-4·A-5) / `agents/orchestrator/ERROR-POLICY.md` EX-13 / `specs/VERIFICATION.md`(POLICY-VERIFY) / `agents/CICD-SETTER.md`(실행) / `specs/DEPLOY-ADAPTER.md`(릴리스 다음 단계) |
| 라벨 | **POLICY-RELEASE** (다른 파일이 본 통합 규율을 이 라벨로 인용) |
| 위치 | `ai-dlc-orchestrator/specs/CICD-RELEASE-ADAPTER.md` |

**왜 어댑터인가**: 우리 시스템(시스템 레벨 조율)과 레포가 보유한 *CI/CD·릴리스 머신*(레포 레벨 자동화) 사이의 *번역·중개 레이어*. 우리는 그 머신을 *대체하지 않고* — 그 게이트·플로우를 인식하고, 우회하지 않고, 결과를 지상검증한다.

### 1.1 적용 조건 (발동 vs 비활성)

본 어댑터는 **레포가 CI/CD 그리고/또는 퍼블리시·릴리스 플로우를 실제로 보유할 때만** 발동한다. 보유 여부는 *레포에서 탐지* 한다 — 추정하지 않는다.

| 탐지 신호 (예시 — 동치 수단 대체 가능) | 시사 |
|---|---|
| CI 설정 파일 존재 (예: 파이프라인·워크플로우 정의 파일) | CI 게이트 보유 → §2 발동 |
| 퍼블리시·배포 잡(job)·스테이지 정의 | 릴리스 플로우 보유 → §4 발동 |
| 레지스트리 좌표를 가진 패키지 매니페스트 (예: 퍼블리시 대상이 명시된 패키지 메타데이터) | 퍼블리시 대상 → §4·§8 발동 |
| 사람 승인 게이트 표식 (예: 수동 승인 스테이지·시각회귀 리뷰 체크) | 사람 게이트 보유 → §3 발동 |

**탐지 결과 아무 신호도 없으면 본 어댑터는 비활성(inert)** — 해당 레포의 사이클에서 §2~§8을 적용하지 않는다. (범용 툴 전제: CI/릴리스가 없는 레포·프로젝트도 정상.) 탐지 결과는 REPO-MAP 또는 사이클 컨텍스트에 기록해 같은 사이클 내 재탐지 비용을 줄인다.

## 2. CI 게이트 인식

오케스트레이터는 변경을 *착지(land)시키기 전에* 레포의 CI 게이트 집합을 **먼저 이해** 한다 — 어떤 게이트가 존재하고, 그중 **무엇이 머지를 차단(BLOCK)하는가**.

### 2.1 게이트 분류

| 게이트 성격 | 예시 (동치 대체 가능) | 처리 |
|---|---|---|
| **필수·차단(required·blocking)** | 통과해야만 머지되는 빌드·린트·테스트, "파이프라인 성공 필수" 설정 | 착지 전 통과 확인. 미통과 시 착지 금지 |
| **비차단·정보성** | 통과 권장이나 머지 차단 안 함 | 결과 관측·기록, 차단은 안 함 |
| **사람 승인 필수** | 수동 수락이 필요한 게이트(예: 시각회귀 리뷰) | §3 별도 처리 (사람 수락 대기) |

탐지 출처(예시): 파이프라인·워크플로우 정의 파일, 브랜치 보호·머지 요건 설정, "이 검사 통과 필수" 표식. 무엇이 *required* 인지 확정하지 못하면 추측하지 말고 사람에게 확인(불확실 시 추측 금지 — §3 공통 규칙).

### 2.2 실패는 신호다 (우회 금지)

- **필수 게이트의 실패는 SIGNAL** — 무언가 통과하지 못했다는 지상 사실이다. 게이트를 *비활성화·삭제·우회* 해 "통과"로 만들지 않는다.
- 실패 게이트는 실제 파이프라인 결과·로그를 읽어(§8) 원인을 진단하고, 그 원인을 고친 뒤 *재실행* 한다.
- 권한·스코프 류 실패(예: 커밋 상태 게시 거부, 잡 재시도 거부)는 게이트 로직 문제가 아니라 *자격증명* 문제일 수 있다 → §6 + EX-13로 분기.

## 3. 사람 승인 게이트 (visual-review 등)

일부 게이트는 자동 통과가 *설계상 불가능* 하다 — **사람의 명시적 수락이 있어야만 통과** 한다. 대표 예(예시): 시각회귀(visual-regression) 리뷰 — 변경된 렌더 결과를 사람이 보고 수락할 때까지 **파이프라인이 의도적으로 실패(FAIL)** 한 상태로 머문다.

### 3.1 처리 시퀀스

```
[1] 사람 승인 게이트 탐지 (예: 시각회귀 리뷰 체크가 "수락 대기"로 실패)
       ↓
[2] 리뷰 산출물·링크 표면화 — 사람이 무엇을 봐야 하는지(예: 스냅샷 diff·리뷰 URL)를 사용자에게 제시
       ↓
[3] 사람 수락 대기 (HITL 게이트 — 오케스트레이터는 임의 진행 금지)
       ↓
[4] 사람 수락 확인 후 게이트 재실행 → 통과 검증(§8)
```

### 3.2 금지

- **게이트 비활성화로 자동 우회 금지.** "실패하니까 끈다"는 *결함* 이다 — 게이트의 목적(사람 눈 검증)을 무력화한다. 시각·UX 클레임은 텍스트 대용물로 대체 불가(POLICY-VERIFY §3 — 시각·UX 행).
- 사람 수락 없이 "통과로 간주" 금지. 수락은 *사람만* 줄 수 있는 입력이다.

> 본 절은 특정 도구가 아니라 *패턴* — "파이프라인이 사람 수락까지 의도적으로 실패하는 게이트" — 을 규정한다. 디자인·시각 리뷰 도구는 모두 예시다.

## 4. 릴리스 플로우 (generic 시퀀스)

레포가 퍼블리시·릴리스 플로우를 보유할 때(§1.1), 릴리스는 다음 *순서를 고정* 한다. 도구·명령은 레포 자신의 컨벤션을 따른다.

```
[1] 버전 범프(version bump) — 레포 컨벤션에 따라 버전 증가
       ↓  (★ 범프는 태깅보다 먼저 착지해야 한다 — 퍼블리시되는 산출물의 버전이 일치하도록)
[2] 기본 브랜치 착지 — 릴리스 착지 방식은 DECISIONS A-3(기본 브랜치 직접 vs MR/PR)로 결정
       ↓
[3] 태그/릴리스 — 레포 자신의 태깅·릴리스 컨벤션 따름
       ↓
[4] CI 퍼블리시 — 파이프라인이 산출물을 레지스트리·엔드포인트로 퍼블리시
       ↓
[5] 산출물 존재 검증 (POLICY-VERIFY) — 레지스트리·엔드포인트에 *해당 버전 산출물이 실재* 하는지 직접 질의. 퍼블리시 성공을 *자기 보고로 가정 금지*.
```

### 4.1 순서 불변 규칙

- **버전 범프는 태깅보다 먼저 착지** — 태그가 가리키는 커밋에 이미 범프된 버전이 들어 있어야 퍼블리시 산출물의 버전과 태그가 일치한다. 순서가 뒤집히면 태그·산출물·버전이 어긋난다.
- **착지 방식은 A-3** — 기본 브랜치 직접 push가 가능한지, 보호 규칙·관례가 MR/PR을 요구하는지에 따른다. *머지·릴리스라는 행위 자체는 외부·비가역 → A-4 컨펌* 이 적용될 수 있다.
- **퍼블리시 성공 ≠ 보고** — [5]를 생략하고 "퍼블리시됨"을 신뢰하는 것은 안티패턴(POLICY-VERIFY §6).

## 5. 생산자→소비자 채택 게이트

공유 컴포넌트의 새 버전은 **퍼블리시되고 §4[5]로 검증된 뒤에만** 소비자가 채택한다.

- **정순 고정 (A-5)**: 생산자 릴리스(퍼블리시 + 검증) → 그 *검증된 버전* 을 소비자가 채택. 소비자 먼저 채택은 *금지* — 아직 존재하지 않는(또는 미검증) 버전을 가리켜 *동작 중인 소비자를 깨뜨린다*.
- DP-5 위상 정렬에서 생산자 노드가 소비자 노드보다 선행한다. 채택 *순서* 자체는 의존 그래프로 자율 결정되나, 생산자 *릴리스* 단계는 A-3·A-4가 컨펌을 요구할 수 있다.
- *Cross-ref*: `specs/SYSTEM-WORKFLOW.md` STEP 9(다중 레포 커밋·머지)의 릴리스 게이트 — 생산자 레포의 릴리스 게이트 통과가 소비자 레포 채택의 선행 제약.
- **배포는 릴리스 다음** — 릴리스가 퍼블리시 + 검증(§4[5])된 *뒤*, 그 검증된 산출물의 *환경 배포* 는 본 명세 밖이다. 배포 규율은 `specs/DEPLOY-ADAPTER.md`(**POLICY-DEPLOY**)가 정의하고, 실행은 `agents/DEPLOY-AGENT.md`가 맡는다(배포·헬스·롤백 실패는 ERROR-POLICY EX-14). 즉 *CI/릴리스 → (핸드오프) → 배포* 의 사슬: 본 명세는 산출물을 *퍼블리시·검증* 까지, 이후 *환경 반영* 은 POLICY-DEPLOY로 넘긴다.

## 6. 자격증명·스코프 사전조건

CI/릴리스 작업은 **특정 스코프 + 역할** 을 가진 자격증명을 필요로 한다. 이들을 *사전조건* 으로 포착한다.

### 6.1 스코프가 필요한 대표 작업 (예시)

| 작업 (예시 — 동치 대체 가능) | 필요 성격 |
|---|---|
| 커밋 상태·체크 게시 | 상태 쓰기 스코프 |
| 실패한 잡 재시도·파이프라인 트리거 | 파이프라인 제어 스코프 |
| 레지스트리·엔드포인트로 퍼블리시 | 퍼블리시(쓰기) 스코프 + 대상 레지스트리 역할 |
| MR/PR에 코멘트·승인 | 리뷰·코멘트 스코프 + 레포 역할 |

### 6.2 실패 진단 (EX-13 연계)

- *forbidden / insufficient-scope / unauthorized* 류 실패는 **EX-13로 진단** — 추정 금지, *실제 에러 본문* 을 읽고 **신원(whoami) → 스코프 → 역할·멤버십** 순으로 부족 층위를 특정한다(스코프 부족 vs 역할 부족 vs 토큰 종류 오류 구분).
- **토큰 발급·회전은 사람 게이트** (자격증명 경계 원칙). 오케스트레이터는 올바르게 스코프된 자격증명을 *사람에게 요청* 하고 대기(EX-13 정책 B/D)한다. 시크릿 값은 `.env` 에서 *읽기만*, 추적 파일에 커밋 금지(§3 공통 정책 POLICY-TRACKING).

## 7. CI 변수·시크릿 사전조건

게이트·플로우가 동작하려면 **필요한 CI 변수/시크릿이 먼저 등록** 되어 있어야 한다. 이를 *사전 점검(pre-flight) 체크리스트 개념* 으로 다룬다(범용 — 구체 변수명은 레포마다 다름).

- 릴리스·퍼블리시 전, 플로우가 의존하는 변수/시크릿(예: 퍼블리시 자격증명·레지스트리 좌표·서명 키 등)이 *레포의 CI 변수·시크릿 저장소에 등록되어 있는지* 확인한다.
- 미등록 변수/시크릿은 게이트·잡 실패의 흔한 근인이다 — 실패를 만나면 *변수 미등록 가능성* 을 진단 항목에 포함한다(권한·스코프 실패와 구분: §6/EX-13).
- 변수/시크릿 *값* 의 등록·회전은 §6과 동일하게 **사람 게이트** — 값을 추적 파일에 커밋하지 않는다.

> 본 절은 "어떤 변수가 필요한지"의 *목록 자체* 가 아니라, *"플로우 발동 전에 필요한 변수·시크릿이 등록되어 있어야 한다"는 사전조건 규율* 을 규정한다. 구체 목록은 각 레포가 자신의 플로우에서 도출한다.

## 8. 검증 (POLICY-VERIFY 연계)

모든 CI/릴리스 결과는 *자기 보고가 아니라 지상검증* 으로 확인한다(VERIFICATION.md POLICY-VERIFY).

| CI/릴리스 클레임 | 지상검증 (무엇을 직접 관측하나) |
|---|---|
| "CI 통과 / 게이트 그린" | *실제 파이프라인 실행* 결과·로그를 읽는다 (보고 아님) — POLICY-VERIFY §3 외부 시스템 행 |
| "퍼블리시됨 / 버전 올림" | *레지스트리·엔드포인트* 에 해당 버전 산출물의 존재를 직접 질의 — POLICY-VERIFY §3 퍼블리시·릴리스 행 |
| "상태 게시됨 / 코멘트 달림" | 게시된 *실제 상태·코멘트* 를 관측 (요청 성공 가정 아님) |
| "권한 됨 / 접근 가능" | *실제 API 에러 본문 + 신원·스코프* 를 읽는다 → EX-13 |
| "시각 리뷰 통과" | *사람 수락* 이라는 실제 입력을 확인 (§3) — 텍스트 대용물로 대체 불가 |

- 릴리스는 외부로 나가고 되돌리기 어려우며 여러 레포·소비자에 파급되므로, 검증 사다리(VERIFICATION.md §5)에서 **최소 L2(독립 검증)** 로 올린다.
- 퍼블리시 검증(§4[5])과 게이트 통과 검증(§2·§3)은 *구현한 주체와 다른 관측* 으로 수행한다 — 같은 층위 재진술은 검증이 아니다.

## 변경 이력

본 명세 v1.0 — **POLICY-RELEASE 신설**: 관리 대상 레포의 CI/CD 게이트·사람 승인 게이트·릴리스/퍼블리시 플로우와의 *조건부 통합 표준*.

근거: 실제 운영에서 (1) 시각회귀 리뷰 게이트, (2) 릴리스(버전 범프→태그→퍼블리시), (3) 자격증명 스코프 실패에서 반복 소모가 발생. 본 어댑터는 이 갭을 *범용·조건부* 로 메운다 — CI/릴리스가 없는 레포에서는 비활성.

가정한 cross-ref (오케스트레이터가 정합 검토 권장):
- 릴리스 게이트의 워크플로우 위치를 **SYSTEM-WORKFLOW.md STEP 9(다중 레포 커밋·머지)** 로 가정 — A-5 특수화의 "STEP 9 릴리스 게이트" 표현과 정합. STEP 9 본문에 릴리스 게이트를 명시 보강할지는 별도 PR 검토.
- 도구·기술 명칭은 모두 예시(e.g.)이며 요구사항이 아니다.

향후 변경은 깃 PR/머지 (원칙 8).
