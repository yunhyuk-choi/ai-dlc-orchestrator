# DEPLOY-ADAPTER.md — 배포 규율 명세 (실행 환경 배포 표준)

> **이 문서는 형식·룰 명세 문서.** 에이전트 룰북 아님.
> 본 명세는 검증된 릴리스·산출물을 *실행 중인 환경(running environment)* 에 배포하는 *조건부 규율* 을 정의한다.
> 본 명세는 **룰만** 정의한다 — 실행 주체는 `DEPLOY-AGENT`(피어 에이전트, 별도)이며, 배포 트리거·검증·롤백의 *조율* 은 오케스트레이터가 인식한다. 본 문서는 *무엇을·어떤 경계로 배포하는지의 룰* 만 제공한다.
> 본 명세는 **조건부** 다 — 레포·시스템이 *실제로 실행 환경에 배포할 때만* 발동하고(§1), 없으면 비활성(inert)이다.
> 도구·기술 명칭(IaC 도구·클라우드 CLI·플랫폼 파이프라인·오케스트레이터·서버리스 등)은 모두 *예시(e.g.)* 일 뿐 요구사항이 아니다. 프로젝트·OS·플랫폼에 맞는 동치 수단으로 대체한다.
> 변경은 깃 PR/머지 (원칙 8) — `specs/CICD-RELEASE-ADAPTER.md`(POLICY-RELEASE) / `specs/VERIFICATION.md`(POLICY-VERIFY) / `agents/orchestrator/DECISIONS.md` §5(A-4·A-5) / `agents/orchestrator/ERROR-POLICY.md` EX-13·EX-14 / `templates/REPO-MAP.template.md` `{REPO_PROFILES}` / `specs/SYSTEM-WORKFLOW.md`(착지 시퀀스)가 본 명세에 연계되므로 변경 시 영향 검토 필수.
> 위치: `ai-dlc-orchestrator/specs/DEPLOY-ADAPTER.md`

> **POLICY-DEPLOY (실행 환경 배포)** — 레포·시스템이 실행 환경에 배포할 때(조건부 발동), 배포는 *환경 모델(낮은 → 높은 티어 승격) + 레포가 포착한 방식에 적응하는 배포 전략* 을 따른다. **non-prod 환경은 에이전트 자율 배포 가능, production(또는 비가역·고영향 타깃)은 사람 게이트(A-4)** — 에이전트는 준비·검증하되 prod 트리거는 *사람의 명시적 승인 후에만*(일반 "계속/진행"은 prod 포괄 승인 아님). 모든 배포는 *실행 환경에 대한 배포 후 지상검증(헬스·스모크, POLICY-VERIFY)* 으로 확인하고, 검증 실패 시 *최종 정상 상태로 자동 롤백*(non-prod 자율 / prod는 복구를 안전 기본값으로 진행하되 사람에게 에스컬레이트)한다. 배포 자격증명·권한 발급은 *사람 게이트*(EX-13).

## 1. 정체성 / 적용 조건 (조건부)

| 항목 | 내용 |
|---|---|
| 종류 | 형식·인터페이스 명세 문서 (에이전트 X) |

| 참조 주체 | `specs/CICD-RELEASE-ADAPTER.md`(POLICY-RELEASE) / `specs/VERIFICATION.md`(POLICY-VERIFY) / `agents/orchestrator/DECISIONS.md` §5(A-4·A-5) / `agents/orchestrator/ERROR-POLICY.md`(EX-13·EX-14) / `templates/REPO-MAP.template.md`(`{REPO_PROFILES}`) |
| 라벨 | **POLICY-DEPLOY** (다른 파일이 본 배포 규율을 이 라벨로 인용) |

**릴리스와의 경계**: `CICD-RELEASE-ADAPTER`(POLICY-RELEASE)는 *"산출물 퍼블리시 + 레지스트리 존재 검증"* 에서 멈춘다. 본 명세는 그 *검증된 산출물을 실행 중인 환경에 올리는* 다음 단계 — 생명주기의 후반부 — 를 규율한다. 둘은 핸드오프로 연결된다(§10).

### 1.1 적용 조건 (발동 vs 비활성)

본 어댑터는 **레포·시스템이 실행 환경에 실제로 배포할 때만** 발동한다. 보유 여부는 *레포에서 탐지* 한다 — 추정하지 않는다.

| 탐지 신호 (예시 — 동치 수단 대체 가능) | 시사 |
|---|---|
| 레포 프로파일(REPO-MAP `{REPO_PROFILES}`)에 배포 대상·환경 정보 존재 | 배포 보유 → §2~§7 발동 |
| 배포 설정·매니페스트 존재 (예: 인프라 정의 파일, 배포 디스크립터, 환경별 설정) | 배포 메커니즘 보유 → §3·§8 발동 |
| 환경 엔드포인트·헬스체크 경로 정의 | 실행 환경 + 배포 후 검증 대상 보유 → §5 발동 |
| 파이프라인에 배포 스테이지 정의 | 배포가 CI/CD 단계 → §10 핸드오프 |

**탐지 결과 아무 신호도 없으면 본 어댑터는 비활성(inert)** — 해당 레포의 사이클에서 §2~§10을 적용하지 않는다. (범용 전제: 라이브러리·산출물만 퍼블리시하고 *실행 환경에 배포하지 않는* 레포·프로젝트도 정상.) 탐지 결과는 REPO-MAP 또는 사이클 컨텍스트에 기록해 같은 사이클 내 재탐지 비용을 줄인다.

## 2. 환경 모델

배포 대상은 **티어(tier)로 계층화된 실행 환경 집합** 이다. 티어 *구성은 프로젝트마다 다르며*, 본 명세는 형태가 아니라 *관계* 를 규정한다.

| 환경 티어 (예시 — 프로젝트별 구성 가능) | 성격 | 본 명세에서의 취급 |
|---|---|---|
| **dev / 통합** | 빠른 반복, 영향 국소, 되돌리기 쉬움 | non-prod → §4 자율 가능 |
| **staging / pre-prod** | prod 근사, 승격 직전 검증대 | non-prod → §4 자율 가능 (단 prod 직전 게이트) |
| **production / live** | 사용자·외부 노출, 비가역·고영향 | prod → §4 **사람 게이트(A-4)** |

- **승격 방향 불변**: 배포는 *낮은 티어 → 높은 티어* 로만 승격한다(§7). 높은 티어에서 검증 없이 시작하지 않는다.
- 티어 *이름·개수* 는 레포 프로파일에서 가져온다 — dev/staging/prod는 *예시* 이며, 단일 환경만 가진 프로젝트도, 다중 stage를 가진 프로젝트도 정상.
- **prod 동치 판정**: "production" 이라는 *이름* 이 아니라 *성격* (사용자·외부 노출·비가역·고영향)으로 prod 게이트(§4) 적용 여부를 판정한다. 이름이 달라도 비가역·고영향 타깃이면 prod로 취급한다.

## 3. 배포 전략

배포 전략은 **레포가 포착한 배포 방식에 적응** 해 선택한다 — 본 명세가 특정 전략을 강제하지 않는다.

| 전략 (예시 — 동치 대체 가능) | 성격 | 적응 신호 |
|---|---|---|
| **롤링(rolling)** | 점진 교체, 무중단 지향 | 다중 인스턴스·무중단 요구 포착 시 |
| **블루-그린(blue-green)** | 병렬 환경 전환, 즉시 롤백 용이 | 환경 복제·트래픽 전환 수단 포착 시 |
| **카나리(canary)** | 일부 트래픽 선노출 후 확대 | 트래픽 분할·점진 노출 수단 포착 시 |
| **재생성(recreate)** | 정지 후 교체 | 무중단 불요·단순 배포 포착 시 |

- 전략은 **레포 프로파일(`{REPO_PROFILES}`)에 포착된 배포 방식** 에서 도출한다 — 포착되지 않았으면 추측하지 말고 사람에게 확인(불확실 시 추측 금지 — 공통 규칙).
- 전략 선택은 *환경 티어에 따라 다를 수 있다* (예: non-prod는 재생성, prod는 카나리). 본 명세는 *전략을 적응적으로 고른다* 는 규율만 규정하고, 구체 전략은 레포가 결정한다.

## 4. ★ 실행 경계 (가장 중요)

본 명세의 핵심 — **배포 자율성의 경계는 환경 티어의 가역성·영향도로 결정된다.**

| 대상 | 자율 vs 게이트 | 규율 |
|---|---|---|
| **non-prod** (dev·staging 등, 되돌리기 쉬움·국소 영향) | **에이전트 자율 배포 가능** | DEPLOY-AGENT는 준비·배포·배포 후 검증(§5)·롤백(§6)까지 자율 수행 |
| **production (또는 비가역·고영향 타깃)** | **사람 게이트 (A-4)** | 에이전트는 *준비·검증* 만 자율 — 실제 prod 트리거는 *사람의 명시적 승인 후에만* |

### 4.1 prod 게이트 규율

- **에이전트는 준비하고 검증한다 — 트리거는 사람이 준다.** prod 배포 직전까지(전략 구성·산출물 확인·staging 검증·롤백 계획)는 자율로 진행하되, *실제 prod 적용 행위* 는 보류하고 사람 승인을 대기한다.
- **일반 "계속/진행"은 prod 배포 포괄 승인이 아니다.** 사이클 전반에 "진행해"가 깔려 있어도, prod 배포라는 *그 특정·비가역·외부 행위* 에 대한 **명시적 승인** 만 자율을 허용한다(A-4 — 일반 위임 ≠ 비가역 외부 행위 포괄 위임).
- prod 배포는 *외부로 나가고 되돌리기 어려운 행위* 이므로 **A-4(외부·비가역 행위 승인)** 의 직접 적용 사례다 (A-4 예시 목록의 `deploy`).
- **자격증명 경계 병존**: 배포 자격증명·권한의 *발급·접근* 은 본 자율 경계와 별개로 항상 사람 게이트다(§9). 즉 non-prod 자율 배포라도 자격증명 *발급* 은 사람이 한다.

> 본 절은 특정 환경 이름이 아니라 *경계 패턴* — "가역·국소(non-prod)는 자율, 비가역·고영향(prod)은 사람 게이트" — 을 규정한다. 환경 이름·개수는 모두 예시다.

## 5. 배포 후 검증 (POLICY-VERIFY 연계)

모든 배포는 *"배포 명령이 exit 0 했다"는 가정이 아니라* **실행 중인 환경에 대한 지상검증** 으로 확인한다(VERIFICATION.md POLICY-VERIFY).

| 배포 클레임 | 지상검증 (무엇을 직접 관측하나) |
|---|---|
| "배포됨 / 적용됨" | *실행 중인 환경* 의 헬스체크·스모크 체크를 직접 호출해 *실제 응답* 을 관측 — POLICY-VERIFY §3 외부 시스템·배포 행 |
| "새 버전 떴음" | 실행 환경이 *실제로 서빙하는 버전·산출물* 을 직접 질의 (배포 도구 보고 아님) |
| "정상 동작" | 핵심 경로 스모크 — *실제 요청·응답 바이트* 로 확인 (프로세스 기동 ≠ 정상 서빙) |
| "롤백됨" | 실행 환경이 *직전 정상 상태* 를 다시 서빙하는지 직접 관측(§6) |

- **배포 = 외부·비가역·파급** 이므로 검증 사다리(VERIFICATION.md §5)에서 **최소 L2(독립 검증)**. prod 배포는 고위험이면 L3(패널·다수결)로 올릴 수 있다.
- 검증은 *배포를 수행한 주체와 다른 관측* 으로 한다 — "배포했다"는 같은 층위 재진술은 검증이 아니다. 배포 도구의 성공 코드를 *지상검증으로 착각 금지*(POLICY-VERIFY §6 안티패턴).
- 배포 후 검증 *실패* 는 SIGNAL — 환경이 정상 상태가 아니라는 지상 사실이다 → §6 롤백 트리거.

## 6. 롤백

배포 후 헬스·검증(§5)이 실패하면, 환경을 **최종 정상 상태(last good state)로 자동 롤백** 한다.

### 6.1 트리거 + 안전 상태 (generic)

- **롤백 트리거**: §5 배포 후 지상검증 실패(헬스·스모크 미통과, 서빙 버전 불일치, 핵심 경로 오류) = 롤백 발동 신호. "배포 도구 성공" 이어도 *환경 검증이 실패* 하면 롤백한다.
- **안전 상태(safe state)**: 배포 직전의 *검증된 정상 상태* — 이전 버전·이전 구성·이전 트래픽 라우팅 등 레포의 배포 방식이 보존하는 복원점. 안전 상태 *형태* 는 전략에 따른다(예: 블루-그린은 이전 환경으로 트래픽 복귀, 롤링은 이전 버전 재배포). 본 명세는 *복원점이 존재하고 그리로 되돌린다* 는 개념만 규정한다.

### 6.2 자율 경계

| 대상 | 롤백 처리 |
|---|---|
| **non-prod** | **자율 롤백** — DEPLOY-AGENT가 안전 상태로 즉시 복원하고 기록 |
| **production** | **롤백은 안전 기본값(safe default)** — 서비스 복원을 위해 진행할 수 있으나, *사람에게 즉시 에스컬레이트*. (서비스 중단 상태를 방치하지 않되, 사후 사람 판단을 보장) |

- prod 롤백은 "복원 = 안전" 이라는 점에서 *전진 배포(고영향 비가역)와 비대칭* — 복원 자체는 진행할 수 있으나 사람 통지·에스컬레이트는 필수다.
- 롤백 자체도 §5로 검증한다 — *되돌렸다는 보고가 아니라* 환경이 실제로 정상 상태를 서빙하는지 관측.
- 롤백 실패·자격증명 부족 등은 **EX-14(배포·롤백 예외)** 로 진단(§11 가정 참조).

## 7. 환경 승격 (promotion)

검증된 릴리스를 **낮은 티어 → 높은 티어** 로 승격한다.

```
[1] 낮은 티어 배포 (예: staging) — non-prod 자율(§4)
       ↓
[2] 배포 후 지상검증 (§5) — 헬스·스모크 통과 확인
       ↓
[3] 승격 게이트 — 높은 티어가 prod(또는 비가역·고영향)면 §4 사람 게이트(A-4)
       ↓
[4] 높은 티어 배포 — 사람 승인 후 트리거 → 다시 §5 검증
```

- **승격 = 검증 전제**: 낮은 티어에서 §5 검증을 통과한 *그* 릴리스만 다음 티어로 승격한다. 미검증 산출물 승격 금지.
- **높은 티어 게이트**: 승격 대상이 prod면 §4 사람 게이트가 승격 트리거에 적용된다.
- 승격 *순서* 는 §2 환경 모델의 티어 위상(낮은 → 높은)을 따른다 — 건너뛰기는 레포가 명시 허용한 경우에만.

---

## 8. IaC·배포 도구 위탁

실제 배포 *메커니즘* 은 **레포가 선택한 도구** 에 위탁한다 — 본 명세는 도구를 *지정하지 않는다*.

- 배포 실행 수단(예: IaC 도구, 클라우드 CLI, 플랫폼 파이프라인, 배포 오케스트레이터)은 레포 프로파일(`{REPO_PROFILES}`)에 포착된 것을 따른다. DEPLOY-AGENT는 그 도구에 *적응* 하며 특정 도구를 *강제하지 않는다*.
- 도구가 무엇이든 본 명세의 *규율* (환경 모델·전략 적응·실행 경계·배포 후 검증·롤백)은 동일하게 적용된다 — 규율은 도구 위에 있는 층이다.
- 배포 도구의 출력·상태는 *클레임* 이며, 환경 지상검증(§5)으로 확인한다 — 도구가 무엇이든 "도구가 성공했다 ≠ 환경이 정상이다".

> 본 절은 "어떤 배포 도구를 쓰라"가 아니라, *"실제 배포 메커니즘은 레포의 선택이고, 에이전트는 그것에 적응한다"는 위탁 규율* 을 규정한다. 나열된 도구는 모두 예시다.

---

## 9. 자격증명·권한 사전조건

배포는 **대상 환경에 대한 자격증명·역할** 을 필요로 한다. 이를 *사전조건* 으로 포착한다.

- 배포 자격증명·역할(예: 환경 접근 키, 배포 권한 롤, 레지스트리 풀 권한)은 **레포 프로파일(`{REPO_PROFILES}`)에 *이름·스코프* 로** 기록된다. **시크릿 *값* 은 `.env` 에서만** 읽고, 프로파일·추적 파일에는 *이름만*(POLICY-TRACKING 준수).
- *forbidden / insufficient-scope / unauthorized* 류 권한 실패는 **ERROR-POLICY EX-13** 로 진단 — 추정 금지, *실제 에러 본문* 을 읽고 **신원(whoami) → 스코프 → 역할·멤버십** 순으로 부족 층위를 특정한다.
- **자격증명·권한의 발급·회전·접근은 사람 게이트** (자격증명 경계 원칙). DEPLOY-AGENT는 올바르게 스코프된 배포 자격증명을 *사람에게 요청* 하고 대기한다(EX-13 정책 B/D). 이는 §4 non-prod 자율 배포와 *별개* — 자율 배포라도 자격증명 발급은 사람이 한다.

## 10. CICD 연계

배포는 **릴리스·퍼블리시가 검증된 뒤(POLICY-RELEASE)** 일어난다.

```
[릴리스] 버전 범프 → 착지 → 태그 → CI 퍼블리시 → 레지스트리 존재 검증   ← POLICY-RELEASE (CICD-RELEASE-ADAPTER)
                                                              │  핸드오프
                                                              ▼
[배포]   환경 모델 → 전략 적응 → (non-prod 자율 / prod A-4) → 배포 후 지상검증 → 롤백   ← POLICY-DEPLOY (본 명세)
```

- **선후 불변**: 배포는 *검증된 산출물* 을 전제한다 — 퍼블리시·레지스트리 검증(POLICY-RELEASE §4[5]) *완료 후* 그 산출물을 환경에 올린다. 미검증 산출물 배포 금지.
- **배포가 CI/CD 스테이지인 경우**: 흔히 배포 자체가 파이프라인의 한 스테이지다. 이때 *파이프라인 배선* 은 CICD 설정 주체(예: CICD-SETTER 에이전트)가 하고, *배포 의미론(환경 경계·검증·롤백)* 은 본 명세(POLICY-DEPLOY)가 규율한다 — 배선과 의미론의 책임 분리.
- 생산자→소비자 채택(A-5)이 *배포* 까지 가는 경우, 생산자 배포·검증 후 소비자 배포 순서를 따른다(§7 승격과 A-5 정합).

## 변경 이력

본 명세 v1.0 — **POLICY-DEPLOY 신설**: 검증된 릴리스를 *실행 중인 환경* 에 배포하는 *조건부 규율* — 환경 모델·적응적 전략·실행 경계(non-prod 자율 / prod 사람 게이트)·배포 후 지상검증·자동 롤백·자격증명 게이트.

근거: `CICD-RELEASE-ADAPTER`(POLICY-RELEASE)가 *"퍼블리시 + 레지스트리 검증"* 에서 멈추므로 *실행 환경 배포* 단계의 규율이 비어 있었다. 본 어댑터가 생명주기 후반부를 *범용·조건부* 로 메운다 — 실행 환경에 배포하지 않는 레포에서는 비활성.

가정한 cross-ref (오케스트레이터가 정합 검토 권장):
- **EX-14(배포·롤백 실패 예외)** — 피어가 ERROR-POLICY에 추가 중인 신규 EX로 가정·참조(§6·§11). 추가 미완 시 본 §6 롤백 실패는 EX-13(권한) 또는 EX-12(검증 실패)로 잠정 분기.
- **`agents/DEPLOY-AGENT.md`** — 본 명세의 실행 주체인 피어 에이전트가 별도 추가될 것으로 가정. 미추가 시 배포 실행 주체는 오케스트레이터가 임시 위탁.
- **SYSTEM-WORKFLOW 착지 시퀀스의 배포 착지 단계** — STEP 9(다중 레포 커밋·머지) 이후 *배포 착지* 를 명시 보강할지는 별도 PR 검토. 현재는 DP-8(종료 직전 외부 착지)과 A-4로 정합.
- 도구·기술 명칭(IaC·클라우드 CLI·플랫폼 파이프라인·배포 전략 등)은 모두 예시(e.g.)이며 요구사항이 아니다.

향후 변경은 깃 PR/머지 (원칙 8).
