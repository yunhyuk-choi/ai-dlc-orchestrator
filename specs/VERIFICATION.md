# VERIFICATION.md — 검증 프로토콜 명세 (지상검증 규율)

> **이 문서는 형식·룰 명세 문서.** 에이전트 룰북 아님.
> 본 명세는 "성공/완료/통과" 보고를 *지상검증(ground truth)* 으로 확인하는 횡단 규율(POLICY-VERIFY)을 정의한다.
> 검증 *실행*은 오케스트레이터 책임 5(결과 통합·검토)와 각 서브 에이전트의 자가 검증. 본 문서는 *무엇을·어떻게 검증하는지의 룰만* 정의.
> 변경은 깃 PR/머지 (원칙 8) — ORCHESTRATOR-AGENT.md §5 / ERROR-POLICY.md(EX 카탈로그) / SYSTEM-WORKFLOW.md §3.1(POLICY-*)이 본 규율에 의존하므로 변경 시 영향 검토 필수.
> 위치: `ai-dlc-orchestrator/specs/VERIFICATION.md`

> **POLICY-VERIFY (검증·지상검증)** — 어떤 주체(서브 에이전트든 자기 자신이든)의 "성공/완료/통과"는 *증거가 아니라 클레임(주장)* 이다. 사용자에게 성공을 보고하거나 의존 작업으로 진행하기 전에, 그 클레임을 *지상검증(가장 낮은 수준의 관찰 가능한 실제 — 상위 보고나 의도가 아님)* 으로 확인한다. 정확성이 중요하거나 되돌리기 어려운 클레임은 *구현자와 다른 독립 검증자* 가 확인한다. 검증 깊이는 위험·비가역성·파급 범위에 비례시킨다.

---

## 1. 정체성

| 항목 | 내용 |
|---|---|
| 종류 | 형식·룰 명세 문서 (에이전트 X) |
| 실행 주체 | 오케스트레이터 (책임 5 결과 통합·검토) + 각 서브 에이전트(자가 검증) + 독립 검증자(정확성 중요 시) |
| 참조 주체 | `agents/orchestrator/ORCHESTRATOR-AGENT.md` §5 / `agents/orchestrator/ERROR-POLICY.md`(검증 미수행 EX) / `specs/SYSTEM-WORKFLOW.md` §3.1(POLICY-*) |
| 라벨 | **POLICY-VERIFY** (다른 파일이 본 규율을 이 라벨로 인용) |
| 위치 | `ai-dlc-orchestrator/specs/VERIFICATION.md` |

**왜 에이전트가 아닌가**: 검증은 *모든 단계에서 발생하는 횡단 규율* — 자가 검증·통합 검토·종료 보고 어디서나 적용. 별도 에이전트로 분리하면 호출 빈도 과다(원칙 1 비효율). 따라서 본 문서는 *룰만* 제공하고, 실행은 각 주체가 직접 수행한다.

---

## 2. 핵심 원칙 — 클레임 ≠ 증거

실제 운영에서 반복 확인된 단일 최대 교훈: **서브 에이전트(및 자기 자신)는 지상 실제가 다른데도 "완료/전부 통과"를 자기 보고하는 일이 반복된다.**

따라서 본 시스템의 검증 규율:

- 주체가 산출한 "성공·완료·통과·해결됨"은 **클레임(주장)이지 증거가 아니다.**
- 사용자에게 성공을 보고하기 전에, *또는* 그 결과에 의존하는 다음 작업으로 진행하기 전에, 클레임을 **지상검증(ground truth)** 으로 확인한다.
- **지상검증 = 가장 낮은 수준의 관찰 가능한 실제.** 상위 요약·중간 보고·작성자의 설명·작성자의 의도가 아니라, 직접 관측되는 최하위 현실을 본다.
- 클레임을 만든 *층위보다 한 단계 아래*를 본다 — 소스가 아니라 산출물, 보고가 아니라 로그, 의도가 아니라 응답 바이트.

## 3. 클레임 유형 → 검증 방법 표

> 도구·기술 명칭은 모두 *예시(e.g.)* 일 뿐 요구사항이 아니다. 프로젝트·OS에 맞는 동치 수단으로 대체한다.

| 클레임 유형 | 클레임 예 | 지상검증 방법 (무엇을 직접 관측하나) |
|---|---|---|
| **코드·파일 변경** | "그 파일 고쳤어 / 반영됨" | 소스나 작성자의 설명이 아니라 *실제 빌드·컴파일·서빙된 산출물* 을 검사한다 (예: 서버가 실제로 내보내는 컴파일 자산의 바이트, 실행 중인 프로세스가 로드한 모듈). |
| **정확성·계산** (수치·접근성·성능·로직) | "값은 N / 대비비 통과 / 응답 X ms" | 주장된 수치를 그대로 받지 말고 *독립적으로 재계산·재측정* 한다. |
| **시각·UX** ("괜찮아 보임", "투박하지 않음") | "레이아웃 깔끔함 / 정렬 맞음" | 사람 눈 또는 *렌더된 스크린샷* 으로 확인한다. 어떤 텍스트 대용물도 시각 검증을 대체하지 못한다. |
| **외부 시스템** (CI·파이프라인·배포) | "CI 통과 / 배포됨" | *실제 실행* 을 트리거·관측하고 그 실제 결과·로그를 읽는다. |
| **퍼블리시·릴리스 산출물** | "패키지 배포했어 / 버전 올림" | *레지스트리·엔드포인트* 에 산출물의 존재·버전을 직접 질의한다. |
| **권한·자격증명 결과** | "토큰 됐어 / 접근 권한 있음" | 추정하지 말고 *실제 API 에러 본문 + 신원(whoami)·스코프* 를 읽는다 (→ ERROR-POLICY 권한/스코프 실패 EX). |

**일반 원칙**: 클레임과 *같은 매체·같은 층위* 에서 나온 진술(작성자가 다시 한 설명, 같은 요약의 다른 표현)은 검증이 아니다. 한 단계 아래의 *독립 관측* 만이 검증이다.

## 4. 독립·대립(adversarial) 검증

정확성이 중요하거나(correctness-critical) 되돌리기 어려운(hard-to-reverse) 클레임에는 *셀프 체크* 로 충분하지 않다 — 구현자는 자기 클레임을 확증하는 방향으로 보기 쉽다(일반적으로 알려진 확증 편향 패턴).

- **검증자는 구현자와 다른 주체여야 한다.** 가능하면 검증자에게 클레임을 *반증(refute)* 하라고 과업을 부여한다 — "맞는지" 가 아니라 "어디서 틀리는지" 를 찾게 한다.
- **고위험(high-stakes)** 결과는 소규모 패널 + *다수결 임계치* 로 판정한다(단일 검증자 1인의 누락 위험 분산).
- 본 절은 특정 과거 사건이 아니라 *일반 패턴* 을 규정한다: 비싼 오류일수록 검증의 독립성·다중성을 높인다.

오케스트레이터 적용: 정확성 중요·비가역 결과를 통합·보고하기 전, *구현한 서브가 아닌 별도 서브(또는 별도 호출)* 를 독립 검증자로 세운다.

---

## 5. 검증 사다리 (escalation)

검증 깊이를 **위험 · 비가역성 · cross-repo/공유 파급(blast radius)** 에 비례시킨다.

| 단계 | 대상 클레임 성격 | 요구 검증 |
|---|---|---|
| **L0 가벼운 확인** | 사소하고 되돌리기 쉬운 변경, 국소적·영향 없음 | 자가 체크 1회(예: 산출물 존재·형식 확인) |
| **L1 지상검증** | 의미 있는 변경, 의존 작업이 뒤따름 | §3 표에 따른 지상검증(소스 아닌 산출물 관측) |
| **L2 독립 검증** | 정확성 중요 또는 비가역 또는 공유·외부 파급 | 지상검증 + §4 독립(가능하면 대립) 검증 |
| **L3 패널·다수결** | 고위험·고비용 오류 | L2 + 소규모 패널 다수결 임계치 |

> 외부로 나가거나(release·deploy·외부 시스템 변경), 되돌리기 어렵거나, 여러 레포·팀원에게 파급되는 클레임은 최소 **L2** 로 올린다. SYSTEM-WORKFLOW.md §3.2 변경·수정의 "Risk ↑ 시 STEP 6 강제" 와 정합.

## 6. 안티패턴 (금지)

- **지상검증 전 성공 보고** — 클레임을 받자마자 사용자에게 "완료" 보고하거나 다음 의존 작업으로 진행.
- **"실행했어" 를 산출물 없이 신뢰** — 실행했다는 *말* 만 받고 실제 결과·로그·산출물을 확인하지 않음.
- **재현 없이 "고쳤다" 선언** — 실패하던 케이스를 다시 돌려보지 않고 수정이 동작한다고 단정.
- **조용한 캡(silent cap)** — 일부만 표본 검사하고 "전수 커버" 라고 주장. 표본 범위·한계를 밝히지 않음.
- **같은 층위 재진술을 검증으로 착각** — 작성자에게 "맞아?" 되묻고 "맞다" 는 답을 검증으로 취급.

위 안티패턴 감지는 ERROR-POLICY.md 의 검증 미수행/실패 EX 로 라우팅한다.

## 7. POLICY-* 연계

본 규율은 SYSTEM-WORKFLOW.md §3.1 공통 정책군(POLICY-*)과 같은 층위의 횡단 정책이며, 그 정책들의 "검증" 요구를 일반화한 단일 원천이다.

| 라벨 | 연계 |
|---|---|
| **POLICY-ENCODING** | "생성 후 각 파일을 검증(U+FFFD·BOM·떠도는 `?` 점검)" 은 *코드·파일 변경* 클레임에 대한 지상검증의 한 사례(§3 1행). |
| **POLICY-TRACKING** | "커밋 성사를 검증" 은 *외부 시스템* 클레임에 대한 지상검증의 한 사례(§3 4행) — 커밋했다는 보고가 아니라 원격 상태를 확인. |
| **POLICY-TEMPLATE-ADHERENCE** | 템플릿 준수 여부는 *인스턴스 산출물* 을 직접 대조해 확인 — 작성자가 "준수했다" 는 진술이 아니라 산출물 관측. |
| **git fetch/pull 선행** (§3) | push 출력의 `[new branch]` 신호를 *실제 출력에서* 읽어 머지·MR 상태를 점검 — 지상검증 규율의 한 적용. |
